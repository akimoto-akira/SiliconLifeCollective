# 安全設計

[English](security.md) | [中文](docs/zh-CN/security.md) | [繁體中文](docs/zh-HK/security.md) | [日本語](docs/ja-JP/security.md) | [한국어](docs/ko-KR/security.md)

## 概述

Silicon Life Collective 的安全性建立在**分层防御**模型之上。核心原則：**所有 I/O 操作必須通過執行器**，執行器在執行前强制執行權限檢查。

```
工具调用 → 執行器 → 權限管理器 → 高拒絕快取 → 高允許快取 → 回调 → 询問使用者
```

---

## 權限模型

### 權限類型

| 類型 | 描述 |
|------|-------------|
| `NetworkAccess` | 出站 HTTP/HTTPS 要求 |
| `CommandLine` | Shell 命令執行 |
| `FileAccess` | 檔案和目錄操作 |
| `Function` | 敏感函式调用 |
| `DataAccess` | 訪問系統或使用者資料 |

### 權限結果

每次權限檢查返回三种結果之一：

| 結果 | 行為 |
|--------|----------|
| **Allowed（允許）** | 操作立即進行 |
| **Denied（拒絕）** | 操作被阻止，記录稽核記錄 |
| **AskUser（询問使用者）** | 操作暂停，需要使用者確認 |

### 特殊角色：硅基主理人

硅基主理人拥有最高權限級别（`IsCurator = true`）。主理人的權限檢查被短路為**允許**，除非使用者明确覆盖。

### 私有權限管理器

每個硅基生命体都有自己的**私有 PermissionManager** 实例。權限狀態不在生命体之间共享。

---

## 權限驗證流程

查詢優先級為：**1. 使用者高拒絕 → 2. 使用者高允許 → 3. 回调函式**

```
┌─────────────┐
│ 工具调用     │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  執行器      │────▶│ 私有權限            │
│（磁碟/網路/  │     │ 管理器（每個生命体）  │
│  命令行...） │     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. IsCurator?   │──是──▶ 允許
                  └────────┬────────┘
                           │ 否
                           ▼
                  ┌─────────────────┐
                  │ 2. 使用者高拒絕    │──匹配──▶ 拒絕
                  │（記憶體快取）      │
                  └────────┬────────┘
                           │ 未匹配
                           ▼
                  ┌─────────────────┐
                  │ 3. 使用者高允許    │──匹配──▶ 允許
                  │（記憶體快取）      │
                  └────────┬────────┘
                           │ 未匹配
                           ▼
                  ┌─────────────────┐
                  │ 4. 權限         │
                  │  回调函式        │──▶ 允許 / 拒絕 / 询問使用者
                  └─────────────────┘
```

**關键点**：執行器只看到布爾值（允許/拒絕）。權限管理器內部處理三元決策（允許/拒絕/询問使用者），並在返回给執行器之前解決询問使用者。

## 執行器（安全邊界）

執行器是 I/O 操作的**唯一**路徑。它們强制執行：

### 独立排程线程

每個執行器拥有**独立的排程线程**：

- 執行器之间的线程隔离 —— 一個執行器的线程阻塞不影響其他執行器。
- 每個執行器可以設定独立的資源限制（CPU、記憶體等）。
- 執行器线程的线程池管理。

### 要求隊列

每個執行器維護一個要求隊列：

- 要求按類型路由到相应的執行器。
- 支援優先級排隊。
- 每個要求的超时控制。

### 權限驗證的线程锁定

當工具發起資源訪問时：

1. 執行器接收要求並**锁定其线程**。
2. 執行器查詢生命体的私有權限管理器。
3. 如果回调返回询問使用者，執行器线程**保持锁定**等待使用者回應。
4. 生命体只看到最终結果（成功或拒絕） —— 它永遠不會看到中间的"待處理"或"等待"狀態。
5. 只有硅基主理人會触發真正的使用者提示。普通生命体同步查詢全局 ACL 而不阻塞。
6. 超时时，要求被视為拒絕，线程锁被释放。

### 執行器類型

| 執行器 | 范围 | 默認超时 |
|----------|-------|-----------------|
| `DiskExecutor` | 檔案读/寫、目錄操作 | 30 秒 |
| `NetworkExecutor` | HTTP 要求、WebSocket 連接 | 60 秒 |
| `CommandLineExecutor` | Shell 命令執行 | 120 秒 |
| `DynamicCompilationExecutor` | Roslyn 記憶體編譯 | 60 秒 |

### 例外隔离與容错

- 一個執行器的例外不影響其他執行器。
- 线程崩溃时自動重啟。
- 熔断器：連续失敗後暂时停止執行器以防止級联失敗。

## 全局 ACL（訪問控制列表）

共享規則表持久化到儲存，僅由硅基主理人管理：

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- 規則按順序評估；第一次匹配获胜。
- 只有硅基主理人可以修改全局 ACL（通過其專用工具）。
- 更改立即生效。
- 全局 ACL **不在**上述每個查詢的優先級链中 —— 它在內部由回调函式引用。

## 使用者频率快取

為減少重复的權限提示，系統維護两個**每個生命体、僅記憶體**的快取：

| 快取 | 用途 |
|-------|---------|
| **HighAllow（高允許）** | 使用者频繁允許的資源 |
| **HighDeny（高拒絕）** | 使用者频繁拒絕的資源 |

### 工作原理

- **使用者选择，非自動检测**：當触發询問使用者时，使用者选择是否将資源添加到快取。
- **前缀匹配**：支援資源路徑前缀匹配（例如 `network:api.example.com/*`）。
- **優先級**：高拒絕的優先級高於高允許。
- **僅記憶體**：快取不持久化。重啟时丟失。
- **可設定過期**：使用者可以設定快取条目的有效期限。

### 快取更新流程

1. 權限回调返回 `AskUser`。
2. 權限系統向卡片系統（Web UI 或即时通訊）發送查詢。
3. 使用者做出決策（允許/拒絕）並**选择是否快取**。
4. 卡片系統返回決策 + 快取标志。
5. 權限系統更新相应的快取列表。
6. 未來匹配快取前缀的要求立即解決。

## 使用者询問機制

當權限檢查返回 `AskUser` 时：

### Web UI：交互式卡片

Web 前端立即顯示**交互式卡片**，顯示：

- 資源類型和路徑
- 操作描述
- 允許 / 拒絕按钮
- 可选的"始终允許" / "始终拒絕"复选框（添加到频率快取）

### 即时通訊（无卡片支援）：随機码

對於不支援交互式卡片的訊息傳递平台：

1. 系統生成两個随機 6 位程式碼：**允許程式碼**和**拒絕程式碼**。
2. 發送包含資源資訊和两個程式碼的訊息。
3. 使用者必須回复准确的允許程式碼才能授權。任何其他回复被视為拒絕。
4. 程式碼是单次使用的，以防止重放攻擊。

### 超时

- 為所有询問使用者要求設定超时。
- 超时时，要求被视為**拒絕**，執行器线程锁被释放。

## 動态編譯安全

自我進化（類別重寫）引入了独特的安全風险。系統使用**分层策略**缓解它們：

### 第 1 层：編譯时引用控制（主要防御）

- 編譯器只获得**允許的装配引用列表**。
- **允許**：`System.Runtime`、`System.Private.CoreLib`、項目装配（ITool 介面等）
- **阻止**：`System.IO`、`System.Reflection`、`System.Runtime.InteropServices` 等
- 如果程式碼引用了被阻止的装配，**編譯器本身拒絕**程式碼。
- 这比執行时掃描更可靠 —— 危險操作在類型級别是不可能的。

### 第 2 层：執行时静态分析（次要防御）

- 即使成功編譯後，程式碼也要進行静态模式掃描。
- 检测危險操作模式（直接 I/O、系統调用等）。
- 如果發現危險程式碼，載入被拒絕，系統回退到默認功能。

### 继承約束

所有自定義硅基生命体類別**必須**继承 `SiliconBeingBase`。編譯器在類型級别强制執行此約束。

### 加密儲存

編譯的程式碼在磁碟上以 AES-256 加密儲存：

- **金鑰派生**：從生命体的 GUID（大寫）使用 PBKDF2。
- **解密失敗**：回退到默認實現。
- **執行时重新編譯**：新程式碼先在記憶體中編譯；只有在成功編譯和实例替换後才持久化。

### 原子替换

替换過程是原子的：

1. 在記憶體中編譯新程式碼 → 获取 `Type`。
2. 從 `Type` 建立新实例。
3. 從舊实例遷移狀態到新实例。
4. 交换引用。
5. 持久化加密程式碼。

如果任何步驟失敗，舊实例保持活動。

## 權限回调函式

### 設計

每個 PermissionManager 持有**回调函式變數**：

- **默認**：指向內置默認權限函式。
- **動态編譯後**：被生命体的自定義權限函式覆盖。
- **二选一**：任何时候只有一個回调處於活動狀態。
- **編譯失敗**：不影響當前回调 —— 默認或上次成功的自定義函式保持有效。

### 回调簽章

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

返回 `Allowed`、`Denied` 或 `AskUser`。

## 稽核記錄

所有權限決策都被記录：

```
[2026-04-01 15:30:25] ALLOWED  | Being:AssistantA | Type:NetworkAccess | Resource:api.github.com | Source:HighAllowCache
[2026-04-01 15:30:26] DENIED   | Being:AssistantB | Type:FileAccess    | Resource:C:\Windows\System32 | Source:HighDenyCache
[2026-04-01 15:30:27] ASK_USER | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:Callback
[2026-04-01 15:30:28] ALLOWED  | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:UserDecision
```

記錄持久化到儲存，可通過 Web UI（記錄控制器）查看。

## Token 使用稽核

`TokenUsageAuditManager` 提供與安全性相關的 AI token 消耗跟踪：

- **每次要求記录** —— 每次 AI 调用記录生命体 ID、模型、提示词 token、补全 token 和時間戳。
- **例外检测** —— 例外的 token 消耗模式可能表明提示词注入或資源滥用。
- **僅主理人訪問** —— `TokenAuditTool`（标記為 `[SiliconManagerOnly]`）允許主理人查詢和汇总 token 使用。
- **Web 儀表板** —— `AuditController` 提供基於瀏覽器的儀表板，带趋势图和資料導出。
- **持久化儲存** —— 記录通過 `ITimeStorage` 儲存，用於時間序列查詢和長期分析。
