# 安全設計

[English](../en/security.md) | [中文](../zh-CN/security.md) | **繁體中文** | [Español](../es-ES/security.md) | [日本語](../ja-JP/security.md) | [한국어](../ko-KR/security.md) | [Čeština](../cs-CZ/security.md)

## 概述

Silicon Life Collective 的安全性建立在**分層防禦**模型之上。核心原則：**所有 I/O 操作必須通過執行器**，執行器在執行前強制執行權限檢查。

```
工具調用 → 執行器 → 權限管理器 → 高拒絕快取 → 高允許快取 → 回呼 → 詢問使用者
```

---

## 權限模型

### 權限類型

| 類型 | 描述 |
|------|-------------|
| `NetworkAccess` | 出站 HTTP/HTTPS 請求 |
| `CommandLine` | Shell 命令執行 |
| `FileAccess` | 檔案和目錄操作 |
| `Function` | 敏感函式調用 |
| `DataAccess` | 存取系統或使用者資料 |

### 權限結果

每次權限檢查返回三種結果之一：

| 結果 | 行為 |
|--------|----------|
| [Deutsch](../de-DE/security.md) | **Allowed（允許）** | 操作立即進行 |
| [Deutsch](../de-DE/security.md) | **Denied（拒絕）** | 操作被阻止，記錄審計日誌 |
| [Deutsch](../de-DE/security.md) | **AskUser（詢問使用者）** | 操作暫停，需要使用者確認 |

### 特殊角色：矽基主理人

矽基主理人擁有最高權限級別（`IsCurator = true`）。主理人的權限檢查被短路為**允許**，除非使用者明確覆寫。

### 私有權限管理器

每個矽基生命體都有自己的**私有 PermissionManager** 實例。權限狀態不在生命體之間共享。

## 權限驗證流程

查詢優先順序為：**1. 使用者高拒絕 → 2. 使用者高允許 → 3. 回呼函式**

### 5 級驗證鏈

所有 AI 發起的操作必須通過完整的 5 級驗證鏈：

```
1. IsCurator（主理人檢查）
   ↓ 如果不是主理人
2. UserFrequencyCache（使用者高頻快取）
   ↓ 如果沒有快取記錄
3. GlobalACL（全域存取控制清單）
   ↓ 如果沒有匹配規則
4. IPermissionCallback（自訂回呼函式）
   ↓ 如果返回 AskUser
5. IPermissionAskHandler（詢問使用者）
```

### 使用者高頻快取

`UserFrequencyCache` 追蹤使用者的常見決策：

- **高允許** - 使用者頻繁允許的規則（跳過 ACL 和回呼）
- **高拒絕** - 使用者頻繁拒絕的規則（立即拒絕）
- **閾值** - 預設 10 次相同決策後進入快取

### 全域存取控制清單（GlobalACL）

`GlobalACL` 管理全域權限規則：

- 基於資源前綴匹配（例如 `/data/`、`http://api.example.com/*`）
- 支援允許/拒絕規則
- 可通過 Web UI 或 `PermissionTool` 管理

## 執行器安全

### 執行器作為唯一 I/O 路徑

所有需要存取外部資源的操作**必須**通過執行器：

- **DiskExecutor** - 檔案系統操作
- **NetworkExecutor** - HTTP/HTTPS 請求
- **CommandLineExecutor** - Shell 命令執行

### 執行器架構

每個執行器擁有：

- **獨立的調度執行緒** - 帶有用於權限驗證的執行緒鎖定
- **請求佇列** - 支援優先順序和超時控制
- **熔斷器** - 連續失敗時暫停執行器
- **審計日誌** - 記錄所有操作

### 異常隔離

執行器之間的故障相互隔離：

- 一個執行器的失敗不影響其他執行器
- 熔斷器防止級聯失敗
- 超時保護防止無限等待

## 動態編譯安全

### 三層防禦機制

矽基生命體的自我進化代碼通過三層安全檢查：

#### 1. 編譯時引用控制（主要防禦）

編譯器只獲得允許的裝配列表：

**允許的裝配：**
- `System.Runtime`
- `System.Collections`
- `System.Linq`
- `System.Threading.Tasks`
- 其他安全的核心庫

**排除的危險裝配：**
- `System.IO` - 檔案系統存取
- `System.Reflection` - 反射
- `System.Net` - 網路存取
- `System.Diagnostics` - 行程控制
- 其他可能危害系統的裝配

這使得危險代碼在類型級別是不可能的。

#### 2. 執行時靜態分析（次要防禦）

`SecurityScanner` 在成功編譯後掃描代碼：

- 檢測危險模式（反射、不安全代碼等）
- 驗證繼承關係（必須繼承 `SiliconBeingBase`）
- 檢查方法簽名

#### 3. 加密儲存

自訂代碼以 AES-256 加密儲存：

- **金鑰派生** - 從生命體 GUID（大寫）通過 PBKDF2 派生
- **加密演算法** - AES-256-CBC
- **完整性檢查** - HMAC-SHA256 驗證

### 代碼替換流程

```
1. AI 生成新類別代碼
   ↓
2. 編譯時引用控制檢查
   ↓ 失敗 → 拋棄代碼
3. Roslyn 記憶體中編譯
   ↓ 失敗 → 拋棄代碼
4. 執行時靜態分析
   ↓ 失敗 → 拋棄代碼
5. 替換當前實例
   ↓
6. 遷移狀態
   ↓
7. 加密儲存到磁碟
```

失敗時，保留現有實現，確保系統穩定性。

## 審計日誌

### 權限決策審計

`AuditLogger` 記錄所有權限決策：

| 欄位 | 描述 |
|------|------|
| `Timestamp` | 決策時間 |
| `BeingId` | 矽基生命體 ID |
| `PermissionType` | 權限類型 |
| `Resource` | 請求的資源 |
| `Result` | 決策結果（Allowed/Denied/AskUser） |
| `Reason` | 決策原因 |
| `ValidationLevel` | 通過的驗證級別（1-5） |

### 審計日誌查詢

審計日誌可通過以下方式查詢：

- **Web UI** - AuditController 儀表板
- **TokenAuditTool** - 僅主理人可用
- **API** - `/api/audit` 端點

## 安全最佳實踐

### 1. 最小權限原則

- 僅授予生命體完成任務所需的最小權限
- 定期審查全域 ACL 規則
- 避免過度使用主理人權限

### 2. 監控和警報

- 定期檢查審計日誌
- 監控權限拒絕率
- 注意異常模式

### 3. 代碼審查

- 審查自訂的 `IPermissionCallback` 實現
- 驗證動態編譯的代碼質量
- 測試邊界情況

### 4. 備份和恢復

- 定期備份生命體數據目錄
- 測試代碼恢復流程
- 保留已知良好的代碼版本

## 常見安全問題

### Q: 如何撤銷生命體的權限？

**A**: 通過以下方式之一：

1. **全域 ACL** - 添加拒絕規則
2. **自訂回呼** - 實現條件邏輯
3. **使用者快取** - 手動清除快取

### Q: 動態編譯的代碼被惡意篡改怎麼辦？

**A**: 三層防禦機制確保安全：

1. 編譯時阻止危險 API 存取
2. 執行時掃描危險模式
3. 加密儲存防止離線篡改

### Q: 如何監控可疑活動？

**A**: 使用審計系統：

- 檢查權限拒絕率異常
- 監控資源存取模式
- 設置自動警報規則
