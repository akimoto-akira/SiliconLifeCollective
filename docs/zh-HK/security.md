# 安全設計

[English](../../security.md) | [简体中文](../zh-CN/security.md) | [日本語](../ja-JP/security.md)

## 概述

矽基生命群的安全體系基於**分層防禦**模型。核心原則：**所有 I/O 操作必須透過執行器**，執行器在執行前強制進行權限驗證。

```
工具呼叫 → 執行器 → 權限管理器 → 高頻拒絕快取 → 高頻允許快取 → 回呼函式 → 詢問使用者
```

---

## 權限模型

### 權限類型

| 類型 | 說明 |
|------|------|
| `NetworkAccess` | 對外 HTTP/HTTPS 請求 |
| `CommandLine` | Shell 命令執行 |
| `FileAccess` | 檔案和目錄操作 |
| `Function` | 敏感函式呼叫 |
| `DataAccess` | 系統或使用者資料存取 |

### 權限判定結果

每次權限檢查返回三種結果之一：

| 結果 | 行為 |
|------|------|
| **Allowed** | 操作立即執行 |
| **Denied** | 操作被阻止，記錄稽核日誌 |
| **AskUser** | 操作暫停，等待使用者確認 |

### 特殊角色：矽基主理人

矽基主理人擁有最高權限等級（`IsCurator = true`）。主理人的權限檢查會被短路為 **Allowed**，除非使用者明確覆寫。

### 私有權限管理器

每個矽基人擁有自己獨立的**私有權限管理器**執行個體。權限狀態不在矽基人之間共用。

---

## 權限驗證流程

查詢優先順序：**1. 使用者高頻拒絕 → 2. 使用者高頻允許 → 3. 權限回呼函式**

```
┌─────────────┐
│ 工具呼叫     │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  執行器     │────▶│ 私有權限管理器      │
│（磁碟/網路/ │     │（每個矽基人獨立）   │
│ 命令列/...）│     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. 是主理人？   │──是──▶ Allowed
                  └────────┬────────┘
                           │ 否
                           ▼
                  ┌─────────────────┐
                  │ 2. 使用者高頻拒絕│──命中──▶ Denied
                  │（記憶體快取）   │
                  └────────┬────────┘
                           │ 未命中
                           ▼
                  ┌─────────────────┐
                  │ 3. 使用者高頻允許│──命中──▶ Allowed
                  │（記憶體快取）   │
                  └────────┬────────┘
                           │ 未命中
                           ▼
                  ┌─────────────────┐
                  │ 4. 權限         │
                  │  回呼函式       │──▶ Allowed / Denied / AskUser
                  └─────────────────┘
```

**關鍵點**：執行器只能看到布林值（Allowed/Denied）。權限管理器內部處理三元決策（Allowed/Denied/AskUser），在 AskUser 流程完成後才將最終布林值返回給執行器。

## 執行器（安全邊界）

執行器是 I/O 操作的**唯一通道**。它們強制執行：

### 獨立排程執行緒

每個執行器擁有**獨立的排程執行緒**：

- 執行器之間的執行緒相互隔離——一個執行器的執行緒阻塞不影響其他執行器。
- 每個執行器可以設定獨立的資源限制（CPU、記憶體等）。
- 使用執行緒集區技術管理執行器執行緒。

### 請求佇列

每個執行器維護一個請求佇列：

- 請求按類型路由到對應的執行器。
- 支援優先順序排隊。
- 每個請求有逾時控制。

### 權限驗證的執行緒鎖定

當工具發起資源存取時：

1. 執行器接收請求並**鎖定當前執行緒**。
2. 執行器查詢該矽基人的私有權限管理器。
3. 如果回呼返回 AskUser，執行器執行緒**保持鎖定**等待使用者響應。
4. 矽基人只能看到最終結果（成功或拒絕）——看不到中間的「暫停」或「等待」狀態。
5. **只有矽基主理人**會觸發真正的使用者詢問。普通矽基人透過全域 ACL 同步查表，不產生阻塞。
6. 逾時後請求視為拒絕，釋放執行緒鎖定。

### 執行器類型

| 執行器 | 範圍 | 預設逾時 |
|--------|------|----------|
| `DiskExecutor` | 檔案讀寫、目錄操作 | 30 秒 |
| `NetworkExecutor` | HTTP 請求、WebSocket 連線 | 60 秒 |
| `CommandLineExecutor` | Shell 命令執行 | 120 秒 |
| `DynamicCompilationExecutor` | Roslyn 記憶體編譯 | 60 秒 |

### 例外隔離與容錯

- 一個執行器的例外不影響其他執行器。
- 執行緒崩潰後自動重新啟動。
- 熔斷機制：連續失敗時暫時熔斷，防止雪崩。

## 全域 ACL（存取控制清單）

持久化到儲存的共用規則表，**僅由矽基主理人管理**：

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- 規則按順序評估，首次比對生效。
- 只有矽基主理人能修改全域 ACL（透過其專用工具）。
- 修改立即生效。
- 全域 ACL **不在**上述每次查詢的優先順序鏈路中——它由回呼函式內部引用。

## 使用者高頻快取

為減少重複的權限提示，系統為每個矽基人維護兩個**僅存在於記憶體中**的快取：

| 快取 | 用途 |
|------|------|
| **高頻允許（HighAllow）** | 使用者頻繁允許的資源 |
| **高頻拒絕（HighDeny）** | 使用者頻繁拒絕的資源 |

### 運作機制

- **使用者主動選擇加入，非系統自動偵測**：當 AskUser 觸發時，使用者選擇是否將資源加入快取。
- **前置詞比對**：支援資源路徑的前置詞比對（如 `network:api.example.com/*`）。
- **優先順序**：高頻拒絕優先順序高於高頻允許。
- **僅記憶體狀態**：快取不持久化，重新啟動後遺失。
- **可設定有效期**：使用者可設定快取項目的有效時長。

### 快取更新流程

1. 權限回呼返回 `AskUser`。
2. 權限系統向卡片系統（Web UI 或 IM）傳送詢問。
3. 使用者做出決策（允許/拒絕）並**選擇是否加入快取**。
4. 卡片系統返回決策 + 快取標記。
5. 權限系統更新對應的快取清單。
6. 後續比對快取前置詞的請求將立即解析。

## 使用者詢問機制

當權限檢查返回 `AskUser` 時：

### Web UI：互動卡片

網頁前端立即彈出**互動卡片**，顯示：

- 資源類型和路徑
- 操作描述
- 允許 / 拒絕按鈕
- 可選的「始終允許」/「始終拒絕」核取方塊（加入高頻快取）

### IM（不支援卡片）：隨機碼

對於不支援互動卡片的即時通訊平台：

1. 系統產生兩個隨機 6 位數字：**允許碼**和**拒絕碼**。
2. 傳送包含資源資訊和兩個碼的訊息。
3. 使用者必須回覆精確的允許碼才能授權。其他任何回覆均視為拒絕。
4. 碼為一次性使用，防止重放攻擊。

### 逾時

- 所有 AskUser 請求都設有逾時時間。
- 逾時後請求視為**拒絕**，釋放執行器執行緒鎖定。

## 動態編譯安全

自我進化（類別重寫）引入了獨特的安全風險。系統透過**分層策略**緩解：

### 第一層：編譯時參考控制（主要防線）

- 編譯器只獲得**允許的組件參考清單**。
- **允許**：`System.Runtime`、`System.Private.CoreLib`、專案組件（ITool 介面等）
- **禁止**：`System.IO`、`System.Reflection`、`System.Runtime.InteropServices` 等
- 程式碼中參考了禁止的組件時，**編譯器本身會報錯並拒絕編譯**
- 這比執行時掃描更可靠——危險操作在類型層面就不可能存在

### 第二層：執行時靜態分析（輔助防線）

- 即使編譯成功，仍對程式碼進行靜態模式掃描。
- 偵測直接 I/O 存取、系統呼叫等危險操作模式。
- 發現危險程式碼時拒絕載入並回退到預設函式。

### 繼承約束

所有自訂矽基人類別**必須**繼承 `SiliconBeingBase`。編譯器在類型層面強制執行此約束。

### 加密儲存

編譯後的程式碼以 AES-256 加密形式儲存在磁碟上：

- **金鑰衍生**：透過 PBKDF2 從矽基人 GUID（全大寫）衍生。
- **解密失敗**：回退到預設實作。
- **執行時重新編譯**：新程式碼先在記憶體中編譯；僅在編譯成功且執行個體替換後才持久化。

### 原子替換

替換過程是原子的：

1. 在記憶體中編譯新程式碼 → 取得 `Type`。
2. 從 `Type` 建立新執行個體。
3. 將舊執行個體狀態遷移到新執行個體。
4. 交換參考。
5. 持久化加密程式碼。

任何步驟失敗，舊執行個體保持活躍。

## 權限回呼函式

### 設計

每個權限管理器持有一個**回呼函式變數**：

- **預設**：指向內建的預設權限函式。
- **動態編譯後**：被矽基人的自訂權限函式覆寫。
- **二選一**：同一時刻只有一個回呼生效。
- **編譯失敗不影響**：保持當前生效的函式（預設函式或上一次編譯成功的自訂函式）。

### 回呼簽章

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

返回 `Allowed`、`Denied` 或 `AskUser`。

## 稽核日誌

所有權限決策都會被記錄：

```
[2026-04-01 15:30:25] ALLOWED  | 矽基人:助手A | 類型:NetworkAccess | 資源:api.github.com | 來源:高頻允許快取
[2026-04-01 15:30:26] DENIED   | 矽基人:助手B | 類型:FileAccess    | 資源:C:\Windows\System32 | 來源:高頻拒絕快取
[2026-04-01 15:30:27] ASK_USER | 矽基人:主理人 | 類型:CommandLine   | 資源:del /f /q *.log | 來源:回呼函式
[2026-04-01 15:30:28] ALLOWED  | 矽基人:主理人 | 類型:CommandLine   | 資源:del /f /q *.log | 來源:使用者決策
```

日誌持久化到儲存，可透過 Web 介面（日誌控制器）查看。

## Token 用量稽核

`TokenUsageAuditManager` 提供與安全相關的 AI Token 消耗追蹤：

- **單次請求記錄** — 每次 AI 呼叫記錄矽基人 ID、模型、提示詞 Token 數、補全 Token 數和時間戳記。
- **異常偵測** — 異常的 Token 消耗模式可能表明提示注入或資源濫用。
- **僅主理人可存取** — `TokenAuditTool`（標記為 `[SiliconManagerOnly]`）允許主理人查詢和彙總 Token 用量。
- **Web 儀表板** — `AuditController` 提供基於瀏覽器的儀表板，含趨勢圖表和資料匯出。
- **持久化儲存** — 記錄透過 `ITimeStorage` 儲存，支援時間序列查詢和長期分析。
