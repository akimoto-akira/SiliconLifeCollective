## 지침 원칙

각 단계는 **실행 가능하고 관찰 가능한** 시스템으로 종료됩니다. "보여줄 것은 없고 인프라만 잔뜩"인 단계는 없습니다.

## ~~단계 1: 채팅 가능~~ ✅ 완료됨

**목표**: 콘솔 입력 → AI 호출 → 콘솔 출력. 최소 검증 단위.

| # | 모듈 | 설명 |

| 1.1 | 솔루션 및 프로젝트 구조 | `SiliconLifeCollective.sln` 생성, `src/SiliconLife.Core/` (코어 라이브러리) 및 `src/SiliconLife.Default/` (기본 구현 + 진입점) 포함 |

## ~~단계 2: 뼈대 있음~~ ✅ 완료됨

**목표**: "맨 루프"를 프레임워크 구조로 교체. 동작은 동일.

| # | 모듈 | 설명 |
|---|--------|-------------|
| 2.1 | 저장 (최소화) | `IStorage` 인터페이스 (Read/Write/Exists/Delete, 키-값 쌍). `FileSystemStorage` 구현. 인스턴스 클래스 (정적 아님). 직접 파일 시스템 접근 —— **AI는 IStorage를 제어할 수 없음** | [Deutsch](../de-DE/roadmap.md) |
| 2.2 | 메인 루프 + 클럭 객체 | 무한 루프, 정확한 클럭 간격 (`Stopwatch` + `Thread.Sleep`). 우선순위 스케줄링 |
| 2.3 | IAIClient 표준화 | `IAIClientFactory` 인터페이스. OllamaClient가 표준 인터페이스 구현하도록 리팩토링 |
| 2.4 | 콘솔 마이그레이션 | `while(true)`를 메인 루프 기반 클럭 객체로 마이그레이션. 단계 1과 동일한 동작 |

**결과물**: 메인 루프가 클럭을 실행하고, 콘솔 채팅이 여전히 작동.

**검증**: 테스트 클럭 객체 등록, 초당 클럭 카운트 출력; 콘솔 채팅이 여전히 작동.

| 3.3 | ContextManager (최소화) | 영혼 파일 + 최근 메시지 연결 → AI 호출 → 응답 얻기. 도구 호출 없음, 지속성 없음 |

**검증**: 콘솔 입력 → 메인 루프 클럭 트리거 → 생명체 처리 (영혼 파일 가이드 동작 포함) → AI 응답. 응답 스타일은 단계 1과 달라야 함.

**목표**: 실리콘 생명체가 채팅뿐만 아니라 작업 수행 가능.

| # | 모듈 | 설명 |
|---|--------|-------------|
| 5.1 | ITool + ToolResult | Name, Description, Execute를 포함하는 `ITool` 인터페이스. Success, Message, Data를 포함하는 `ToolResult` |
| 5.2 | ToolManager | 생명체별 인스턴스. 리플렉션 기반 도구 발견. `[SiliconManagerOnly]` 속성 지원 |
| 5.3 | IAIClient: 도구 호출 지원 | AI tool_calls 구문 분석. 루프: 도구 실행 → 결과 다시 전송 → AI 계속 → 순수 텍스트까지 |
| 5.4 | 실행기 베이스 클래스 | 독립적인 스케줄링 스레드, 요청 큐, 시간 초과 제어를 갖춘 추상 베이스 클래스 |
| 5.5 | NetworkExecutor | 실행기를 통한 HTTP 요청. 시간 초과, 큐잉 |
| 5.6 | CommandLineExecutor | 실행기를 통한 셸 실행. 크로스 플랫폼 구분자 감지 |
| 5.7 | DiskExecutor | 실행기를 통한 파일 작업. 권한 검사 아직 없음 (단계 6) |
| 5.8–5.12 | 내장 도구 | CalendarTool, SystemTool, NetworkTool, ChatTool, DiskTool |

**결과물**: 실리콘 생명체가 도구 호출을 통해 작업 수행 가능.

**검증**: "오늘 무슨 요일이야?" 물어보기 → CalendarTool 답변; "프로세스 확인해줘" → SystemTool 실행; 생명체에게 다른 생명체에게 메시지 보내라고 지시 → ChatTool 작동.

## ~~단계 6: 규칙 준수 (권한 시스템)~~ ✅ 완료됨

**목표**: 실리콘 생명체가 승인 없이 민감한 리소스에 접근할 수 없음.

| # | 모듈 | 설명 |
|---|--------|-------------|
| 6.1 | PermissionManager | 생명체별 프라이빗 인스턴스. 콜백 기반, 3항 결과 (허용/거부/사용자에게 문의). 쿼리 우선순위: HighDeny → HighAllow → Callback. IsCurator 플래그 |
| 6.2 | PermissionType 열거형 | NetworkAccess, CommandLine, FileAccess, Function, DataAccess |
| 6.3 | DefaultPermissionCallback | 네트워크 화이트리스트/블랙리스트, CLI 분류, 파일 경로 보안 규칙 |
| 6.4 | GlobalACL | 접두사 일치 규칙 테이블, 저장소에 지속 |
| 6.5 | UserFrequencyCache | HighAllow/HighDeny 목록. 사용자 선택 (자동 감지 아님). 접두사 일치, 메모리 전용, 구성 가능한 만료 |
| 6.6 | UserAskMechanism (콘솔) | AskUser 반환 시 콘솔에서 y/n 프롬프트 |
| 6.7 | 실행기 권한 통합 | 모든 실행기가 실행 전 권한 검사 |
| 6.8 | IStorage 분리 설명 | IStorage는 시스템 내부 지속성 —— 직접 파일 접근, 실행기를 통해 라우팅 **안됨**, AI가 제어 **불가**. 실행기는 AI 도구에서 시작한 IO만 관리 |
| 6.9 | 감사 로그 | 타임스탬프, 요청자, 리소스, 결과가 포함된 모든 권한 결정 기록 |

**결과물**: 생명체가 민감한 작업 시도 시 권한 프롬프트 표시.

**검증**: 생명체에게 파일 삭제 지시 → 콘솔에 권한 프롬프트 표시 → `n` 입력 → 작업 거부. 생명체에게 화이트리스트 웹사이트 접근 지시 → 즉시 허용.
