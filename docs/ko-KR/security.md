# 보안 설계

> **버전: v0.1.0-alpha**

[English](../en/security.md) | [中文](../zh-CN/security.md) | [繁體中文](../zh-HK/security.md) | [Español](../es-ES/security.md) | [日本語](../ja-JP/security.md) | **한국어** | [Čeština](../cs-CZ/security.md)

## 개요

Silicon Life Collective의 보안은 **다층 방어** 모델 위에 구축됩니다. 핵심 원칙: **모든 I/O 작업은 실행기를 통과해야 하며**, 실행기는 실행 전 권한 검사를 강제합니다.

```
도구 호출 → 실행기 → 권한 관리자 → 고거부 캐시 → 고허용 캐시 → 콜백 → 사용자에게 문의
```

---

## 권한 모델

### 권한 타입

| 타입 | 설명 |
|------|-------------|
| `NetworkAccess` | 아웃바운드 HTTP/HTTPS 요청 |
| `CommandLine` | 셸 명령 실행 |
| `FileAccess` | 파일 및 디렉토리 작업 |
| `Function` | 민감한 함수 호출 |
| `DataAccess` | 시스템 또는 사용자 데이터 접근 |

### 권한 결과

각 권한 검사는 세 가지 결과 중 하나를 반환합니다:

| 결과 | 동작 |
|--------|----------|
| [Deutsch](../de-DE/security.md) | **Allowed (허용)** | 작업 즉시 진행 |
| [Deutsch](../de-DE/security.md) | **Denied (거부)** | 작업 차단, 감사 로그 기록 |
| [Deutsch](../de-DE/security.md) | **AskUser (사용자에게 문의)** | 작업 일시중지, 사용자 확인 필요 |

### 특수 역할: 실리콘 큐레이터

실리콘 큐레이터는 최고 권한 레벨(`IsCurator = true`)을 가집니다. 큐레이터의 권한 검사는 사용자가 명시적으로 재정의하지 않는 한 **허용**으로 단락됩니다.

### 전용 권한 관리자

각 실리콘 생명체는 자체 **전용 PermissionManager** 인스턴스를 가집니다. 권한 상태는 생명체 간에 공유되지 않습니다.

## 권한 검증 프로세스

쿼리 우선순위: **1. 사용자 고거부 → 2. 사용자 고허용 → 3. 콜백 함수**

```
┌─────────────┐
│ 도구 호출    │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  실행기      │────▶│ 전용 권한           │
│(디스크/네트워크│     │ 관리자 (각 생명체)   │
│  명령줄...)  │     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. IsCurator?   │──예──▶ 허용
                  └────────┬────────┘
                           │ 아니오
                           ▼
                  ┌─────────────────┐
                  │ 2. 사용자 고거부 │──일치──▶ 거부
                  │(메모리 캐시)     │
                  └────────┬────────┘
                           │ 불일치
                           ▼
                  ┌─────────────────┐
                  │ 3. 사용자 고허용 │──일치──▶ 허용
                  │(메모리 캐시)     │
                  └────────┬────────┘
                           │ 불일치
                           ▼
                  ┌─────────────────┐
                  │ 4. 권한         │
                  │  콜백 함수       │──▶ 허용 / 거부 / 사용자에게 문의
                  └─────────────────┘
```

**핵심 사항**: 실행기는 불리언 값(허용/거부)만 봅니다. 권한 관리자는 내부적으로 삼항 결정(허용/거부/사용자에게 문의)을 처리하며, 실행기에 반환하기 전에 사용자에게 문의를 해결합니다.

## 실행기 (보안 경계)

실행기는 I/O 작업의 **유일** 경로입니다. 다음을 강제합니다:

### 독립 스케줄링 스레드

각 실행기는 **독립 스케줄링 스레드**를 가집니다:

- 실행기 간 스레드 격리 — 한 실행기의 스레드 블로킹이 다른 실행기에 영향을 미치지 않음.
- 각 실행기는 독립적인 리소스 제한 설정 가능 (CPU, 메모리 등).
- 실행기 스레드의 스레드 풀 관리.

### 요청 큐

각 실행기는 요청 큐를 유지합니다:

- 요청은 타입에 따라 해당 실행기로 라우팅.
- 우선순위 큐 지원.
- 요청별 타임아웃 제어.

### 권한 검증의 스레드 락

도구가 리소스 접근을 시작할 때:

1. 실행기가 요청을 받고 **스레드를 락**.
2. 실행기가 생명체의 전용 권한 관리자에 쿼리.
3. 콜백이 사용자에게 문의를 반환하면, 실행기 스레드는 사용자 응답을 기다리며 **락 유지**.
4. 생명체는 최종 결과(성공 또는 거부)만 봅니다 — 중간 "대기 중" 또는 "기다리는" 상태를 보지 못합니다.
5. 실리콘 큐레이터만 실제 사용자 프롬프트를 트리거합니다. 일반 생명체는 블로킹 없이 글로벌 ACL을 동기적으로 쿼리합니다.
6. 타임아웃 시, 요청은 거부로 처리되고 스레드 락이 해제됩니다.

### 실행기 타입

| 실행기 | 범위 | 기본 타임아웃 |
|----------|-------|-----------------|
| `DiskExecutor` | 파일 읽기/쓰기, 디렉토리 작업 | 30초 |
| `NetworkExecutor` | HTTP 요청, WebSocket 연결 | 60초 |
| `CommandLineExecutor` | 셸 명령 실행 | 120초 |
| `DynamicCompilationExecutor` | Roslyn 메모리 컴파일 | 60초 |

### 예외 격리 및 내고장성

- 한 실행기의 예외가 다른 실행기에 영향을 미치지 않음.
- 스레드 충돌 시 자동 재시작.
- 서킷 브레이커: 연속 실패 후 실행기 일시 중지로 연쇄 실패 방지.

## 글로벌 ACL (접근 제어 목록)

저장소에 영속화된 공유 규칙 테이블, 실리콘 큐레이터만 관리:

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- 규칙은 순서대로 평가; 첫 번째 일치가 승리.
- 실리콘 큐레이터만 글로벌 ACL 수정 가능 (전용 도구 통해).
- 변경사항 즉시 적용.
- 글로벌 ACL은 위 각 쿼리의 우선순위 체인에 **없음** — 콜백 함수에서 내부적으로 참조.

## 사용자 주파수 캐시

반복적인 권한 프롬프트를 줄이기 위해, 시스템은 두 개의 **생명체별, 메모리 전용** 캐시를 유지합니다:

| 캐시 | 용도 |
|-------|---------|
| [Deutsch](../de-DE/security.md) | **HighAllow (고허용)** | 사용자가 자주 허용한 리소스 |
| [Deutsch](../de-DE/security.md) | **HighDeny (고거부)** | 사용자가 자주 거부한 리소스 |

### 작동 방식

- **사용자 선택, 자동 감지 아님**: 사용자에게 문의가 트리거될 때, 사용자가 리소스를 캐시에 추가할지 선택.
- **접두사 매칭**: 리소스 경로 접두사 매칭 지원 (예: `network:api.example.com/*`).
- **우선순위**: 고거부가 고허용보다 우선순위 높음.
- **메모리 전용**: 캐시는 영속화되지 않음. 재시작 시 손실.
- **구성 가능한 만료**: 사용자가 캐시 항목의 유효 기간 설정 가능.

### 캐시 업데이트 프로세스

1. 권한 콜백이 `AskUser` 반환.
2. 권한 시스템이 카드 시스템(Web UI 또는 인스턴트 메시징)에 쿼리 전송.
3. 사용자가 결정(허용/거부)을 내리고 **캐시 여부 선택**.
4. 카드 시스템이 결정 + 캐시 플래그 반환.
5. 권한 시스템이 해당 캐시 목록 업데이트.
6. 향후 캐시 접두사와 일치하는 요청 즉시 해결.

## 사용자 문의 메커니즘

권한 검사가 `AskUser`를 반환할 때:

### Web UI: 인터랙티브 카드

웹 프론트엔드가 즉시 **인터랙티브 카드**를 표시:

- 리소스 타입 및 경로
- 작업 설명
- 허용 / 거부 버튼
- 선택적 "항상 허용" / "항상 거부" 체크박스 (주파수 캐시에 추가)

### 인스턴트 메시징 (카드 지원 없음): 랜덤 코드

인터랙티브 카드를 지원하지 않는 메시징 플랫폼의 경우:

1. 시스템이 두 개의 6자리 랜덤 코드 생성: **허용 코드** 및 **거부 코드**.
2. 리소스 정보와 두 코드를 포함한 메시지 전송.
3. 사용자가 정확한 허용 코드로 응답해야 승인. 다른 모든 응답은 거부로 처리.
4. 코드는 일회용으로, 재생 공격 방지.

### 타임아웃

- 모든 사용자에게 문의 요청에 타임아웃 설정.
- 타임아웃 시, 요청은 **거부**로 처리되고 실행기 스레드 락 해제.

## 동적 컴파일 보안

자가 진화(클래스 재작성)는 독특한 보안 위험을 초래합니다. 시스템은 **계층적 전략**으로 이를 완화합니다:

### 1단계: 컴파일 타임 참조 제어 (주요 방어)

- 컴파일러는 **허용된 어셈블리 참조 목록**만 받음.
- **허용**: `System.Runtime`, `System.Private.CoreLib`, 프로젝트 어셈블리 (ITool 인터페이스 등)
- **차단**: `System.IO`, `System.Reflection`, `System.Runtime.InteropServices` 등
- 코드가 차단된 어셈블리를 참조하면, **컴파일러 자체가 코드 거부**.
- 런타임 스캔보다 더 안정적 — 위험한 작업이 타입 레벨에서 불가능.

### 2단계: 런타임 정적 분석 (보조 방어)

- 성공적인 컴파일 후에도, 코드는 정적 패턴 스캔 대상.
- 위험한 작업 패턴 감지 (직접 I/O, 시스템 호출 등).
- 위험한 코드 발견 시, 로딩 거부, 시스템이 기본 기능으로 폴백.

### 상속 제약

모든 맞춤형 실리콘 생명체 클래스는 `SiliconBeingBase`를 **상속해야 합니다**. 컴파일러가 타입 레벨에서 이 제약을 강제합니다.

### 암호화 저장

컴파일된 코드는 디스크에 AES-256으로 암호화되어 저장:

- **키 파생**: PBKDF2를 사용하여 생명체 GUID (대문자)에서 파생.
- **복호화 실패**: 기본 구현으로 폴백.
- **런타임 재컴파일**: 새 코드는 먼저 메모리에서 컴파일; 성공적인 컴파일 및 인스턴스 교체 후에만 영속화.

### 원자적 교체

교체 프로세스는 원자적입니다:

1. 메모리에서 새 코드 컴파일 → `Type` 획득.
2. `Type`에서 새 인스턴스 생성.
3. 이전 인스턴스에서 새 인스턴스로 상태 마이그레이션.
4. 참조 교환.
5. 암호화된 코드 영속화.

어떤 단계든 실패하면, 이전 인스턴스가 활성 상태 유지.

## 권한 콜백 함수

### 설계

각 PermissionManager는 **콜백 함수 변수**를 보유:

- **기본**: 내장 기본 권한 함수를 가리킴.
- **동적 컴파일 후**: 생명체의 맞춤형 권한 함수로 재정의.
- **둘 중 하나**: 언제든지 하나의 콜백만 활성.
- **컴파일 실패**: 현재 콜백에 영향 없음 — 기본 또는 마지막으로 성공한 맞춤형 함수 유지.

### 콜백 시그니처

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

`Allowed`, `Denied` 또는 `AskUser` 반환.

## 감사 로그

모든 권한 결정이 기록됩니다:

```
[2026-04-01 15:30:25] ALLOWED  | Being:AssistantA | Type:NetworkAccess | Resource:api.github.com | Source:HighAllowCache
[2026-04-01 15:30:26] DENIED   | Being:AssistantB | Type:FileAccess    | Resource:C:\Windows\System32 | Source:HighDenyCache
[2026-04-01 15:30:27] ASK_USER | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:Callback
[2026-04-01 15:30:28] ALLOWED  | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:UserDecision
```

로그는 저장소에 영속화되며, Web UI (로그 컨트롤러)를 통해 볼 수 있습니다.

## Token 사용 감사

`TokenUsageAuditManager`는 보안 관련 AI token 소모 추적을 제공합니다:

- **요청별 기록** — 각 AI 호출이 생명체 ID, 모델, 프롬프트 token, 완성 token 및 타임스탬프 기록.
- **이상 감지** — 비정상적인 token 소모 패턴은 프롬프트 주입 또는 리소스 남용을 나타낼 수 있음.
- **큐레이터 전용 접근** — `TokenAuditTool` (`[SiliconManagerOnly]` 표시)은 큐레이터가 token 사용 쿼리 및 집계 가능.
- **웹 대시보드** — `AuditController`는 추세 차트 및 데이터 내보내기가 포함된 브라우저 기반 대시보드 제공.
- **영속화 저장** — 기록은 `ITimeStorage`를 통해 저장되어, 시계열 쿼리 및 장기 분석 가능.
