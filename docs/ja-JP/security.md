# セキュリティ設計

[中文](../zh-CN/security.md) | [繁體中文](../zh-HK/security.md) | [English](../security.md)

## 概要

Silicon Life Collectiveのセキュリティは**多層防御**モデルに基づいて構築されています。基本原則: **すべてのI/O操作はエグゼキューターを通過する必要があり**、エグゼキューターは実行前にパーミッションチェックを強制します。

```
ツールコール → エグゼキューター → PermissionManager → HighDenyキャッシュ → HighAllowキャッシュ → コールバック → AskUser
```

---

## パーミッションモデル

### パーミッションタイプ

| タイプ | 説明 |
|------|-------------|
| `NetworkAccess` | アウトバウンドHTTP/HTTPSリクエスト |
| `CommandLine` | シェルコマンド実行 |
| `FileAccess` | ファイルおよびディレクトリ操作 |
| `Function` | 機密関数呼び出し |
| `DataAccess` | システムまたはユーザーデータへのアクセス |

### パーミッション結果

すべてのパーミッションチェックは3つの結果のいずれかを返します:

| 結果 | 動作 |
|--------|----------|
| **Allowed** | 操作は直ちに実行 |
| **Denied** | 操作はブロック、監査ログ記録 |
| **AskUser** | 操作は一時停止、ユーザー確認が必要 |

### 特別なロール: シリコンキュレーター

シリコンキュレーターは最高権限レベル(`IsCurator = true`)を保持します。キュレーターのパーミッションチェックは、ユーザーによって明示的にオーバーライドされない限り、**Allowed**にショートサーキットされます。

### プライベートパーミッションマネージャー

各シリコンビーイングは独自の**プライベートPermissionManager**インスタンスを持ちます。パーミッション状態はビーイング間で共有されません。

## パーミッション検証フロー

クエリ優先度は: **1. ユーザーHighDeny → 2. ユーザーHighAllow → 3. コールバック関数**

```
┌─────────────┐
│ ツールコール   │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  エグゼキューター │────▶│ プライベートパーミッション  │
│ (ディスク/ネット/  │     │ マネージャー(ビーイングごと) │
│  CLI/...)   │     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. IsCurator?   │──はい──▶ Allowed
                  └────────┬────────┘
                           │ いいえ
                           ▼
                  ┌─────────────────┐
                  │ 2. ユーザーHighDeny│──一致──▶ Denied
                  │ (メモリキャッシュ)  │
                  └────────┬────────┘
                           │ 不一致
                           ▼
                  ┌─────────────────┐
                  │ 3. ユーザーHighAllow│──一致──▶ Allowed
                  │ (メモリキャッシュ)  │
                  └────────┬────────┘
                           │ 不一致
                           ▼
                  ┌─────────────────┐
                  │ 4. パーミッション   │
                  │  コールバック関数  │──▶ Allowed / Denied / AskUser
                  └─────────────────┘
```

**重要ポイント**: エグゼキューターはブール値(Allowed/Denied)のみを表示します。PermissionManagerは内部的に3方向の決定(Allowed/Denied/AskUser)を処理し、エグゼキューターに戻る前にAskUserを解決します。

## エグゼキューター(セキュリティ境界)

エグゼキューターはI/O操作の**唯一**のパスです。以下を強制:

### 独立スケジューリングスレッド

各エグゼキューターは**独立スケジューリングスレッド**を所有:

- スレッドはエグゼキューター間で分離 — 1つのエグゼキューターのスレッドブロックが他のエグゼキューターに影響を与えない。
- 各エグゼキューターは独立したリソース制限(CPU、メモリなど)を設定可能。
- エグゼキュータースレッドのスレッドプール管理。

### リクエストキュー

各エグゼキューターはリクエストキューを維持:

- リクエストはタイプによって対応するエグゼキューターにルーティング。
- 優先度キューイングをサポート。
- リクエストごとのタイムアウト制御。

### パーミッション検証用のスレッドロック

ツールがリソースアクセスを開始した場合:

1. エグゼキューターはリクエストを受信し、**スレッドをロック**。
2. エグゼキューターはビーイングのプライベートPermissionManagerにクエリを実行。
3. コールバックがAskUserを返した場合、エグゼキュータースレッドはユーザーの応答を待って**ロックされたまま**。
4. ビーイングは最終結果(成功または拒否)のみを表示 — 中間の「保留中」または「待機中」状態は表示されない。
5. シリコンキュレーターのみが実際のユーザープロンプトをトリガー。通常のビーイングはブロッキングなしでグローバルACLを同期的にクエリ。
6. タイムアウト時、リクエストは拒否として扱われ、スレッドロックが解放。

### エグゼキュータータイプ

| エグゼキューター | スコープ | デフォルトタイムアウト |
|----------|-------|-----------------|
| `DiskExecutor` | ファイル読み取り/書き込み、ディレクトリ操作 | 30秒 |
| `NetworkExecutor` | HTTPリクエスト、WebSocket接続 | 60秒 |
| `CommandLineExecutor` | シェルコマンド実行 | 120秒 |
| `DynamicCompilationExecutor` | Roslynインメモリコンパイル | 60秒 |

### 例外分離とフォールトトレランス

- 1つのエグゼキューターの例外が他のエグゼキューターに影響を与えない。
- スレッドクラッシュ時の自動再起動。
- サーキットブレーカー: カスケード障害を防ぐために、連続失敗後にエグゼキューターを一時的に停止。

## グローバルACL(アクセス制御リスト)

シリコンキュレーターによって排他的に管理されるストレージに永続化される共有ルールテーブル:

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- ルールは順番に評価; 最初の一致が優先。
- シリコンキュレーターのみがグローバルACLを変更可能(専用ツール経由)。
- 変更は直ちに有効。
- グローバルACLは上記のクエリ優先度チェーンには**ない** — 内部的にコールバック関数によって参照される。

## ユーザー頻度キャッシュ

繰り返しパーミッションプロンプトを減らすために、システムは2つの**ビーイングごと、メモリのみ**のキャッシュを維持:

| キャッシュ | 目的 |
|-------|---------|
| **HighAllow** | ユーザーが頻繁に許可したリソース |
| **HighDeny** | ユーザーが頻繁に拒否したリソース |

### 動作方法

- **ユーザーオプトイン、自動検出ではない**: AskUserがトリガーされたとき、ユーザーがリソースをキャッシュに追加するかどうかを選択。
- **プレフィックスマッチング**: リソースパスプレフィックスマッチングをサポート(例: `network:api.example.com/*`)。
- **優先度**: HighDenyはHighAllowより高い優先度。
- **メモリのみ**: キャッシュは永続化されない。再起動時に失われる。
- **有効期限設定可能**: ユーザーはキャッシュエントリの有効期間を設定可能。

### キャッシュ更新フロー

1. パーミッションコールバックが`AskUser`を返す。
2. パーミッションシステムはカードシステム(Web UIまたはIM)に問い合わせを送信。
3. ユーザーは決定(許可/拒否)を行い、**キャッシュするかどうかを選択**。
4. カードシステムは決定 + キャッシュフラグを返す。
5. パーミッションシステムは対応するキャッシュリストを更新。
6. キャッシュされたプレフィックスに一致する将来のリクエストは直ちに解決。

## ユーザーAskメカニズム

パーミッションチェックが`AskUser`を返した場合:

### Web UI: インタラクティブカード

Webフロントエンドはすぐに**インタラクティブカード**を表示:

- リソースタイプとパス
- 操作の説明
- 許可 / 拒否 ボタン
- オプションの「常に許可」/「常に拒否」チェックボックス(頻度キャッシュに追加)

### IM(カードサポートなし): ランダムコード

インタラクティブカードをサポートしないメッセージングプラットフォームの場合:

1. システムは2つの6桁のランダムコードを生成: **許可コード**と**拒否コード**。
2. リソース情報と両方のコードを含むメッセージを送信。
3. ユーザーは許可を承認するために正確な許可コードを返信する必要があります。他の返信は拒否として扱われる。
4. コードは再利用攻撃を防ぐために一度きり使用。

### タイムアウト

- すべてのAskUserリクエストにタイムアウトが設定。
- タイムアウト時、リクエストは**拒否**として扱われ、エグゼキュータースレッドロックが解放。

## 動的コンパイルセキュリティ

自己進化(クラス書き換え)は固有のセキュリティリスクを導入します。システムは**多層戦略**でそれらを軽減:

### レイヤー1: コンパイル時参照制御(主要防御)

- コンパイラーには**許可されたアセンブリ参照リスト**のみが与えられる。
- **許可**: `System.Runtime`、`System.Private.CoreLib`、プロジェクトアセンブリ(IToolインターフェースなど)
- **ブロック**: `System.IO`、`System.Reflection`、`System.Runtime.InteropServices`など
- コードがブロックされたアセンブリを参照している場合、**コンパイラー自身がコードを拒否**。
- これは実行時スキャンよりも信頼性が高い — 危険な操作はタイプレベルで不可能。

### レイヤー2: 実行時静的分析(二次防御)

- コンパイル成功後でも、コードは静的パターンスキャンを受ける。
- 危険な操作パターン(直接I/O、システムコールなど)を検出。
- 危険なコードが見つかった場合、読み込みは拒否され、システムはデフォルト関数にフォールバック。

### 継承制約

すべてのカスタムシリコンビーイングクラスは**必ず**`SiliconBeingBase`を継承する必要があります。コンパイラーはこれをタイプレベルで強制。

### 暗号化ストレージ

コンパイルされたコードはディスク上でAES-256暗号化で保存:

- **鍵導出**: ビーイングのGUID(大文字)からのPBKDF2。
- **復号化失敗**: デフォルト実装にフォールバック。
- **実行時再コンパイル**: 新しいコードはまずメモリ内でコンパイル; インスタンス交換成功後にのみ永続化。

### アトミック置換

置換プロセスはアトミック:

1. 新しいコードをメモリ内でコンパイル → `Type`を取得。
2. `Type`から新しいインスタンスを作成。
3. 古いインスタンスから新しいインスタンスに状態を移行。
4. 参照をスワップ。
5. 暗号化されたコードを永続化。

いずれかのステップが失敗した場合、古いインスタンスがアクティブなまま。

## パーミッションコールバック関数

### 設計

各PermissionManagerは**コールバック関数変数**を保持:

- **デフォルト**: 組み込みデフォルトパーミッション関数を指す。
- **動的コンパイル後**: ビーイングのカスタムパーミッション関数によってオーバーライド。
- **二者択一**: 一度にアクティブなコールバックは1つのみ。
- **コンパイル失敗**: 現在のコールバックに影響しない — デフォルトまたは最後の成功したカスタム関数が有効なまま。

### コールバックシグネチャ

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

`Allowed`、`Denied`、または`AskUser`を返す。

## 監査ログ

すべてのパーミッション決定がログに記録:

```
[2026-04-01 15:30:25] ALLOWED  | ビーイング:AssistantA | タイプ:NetworkAccess | リソース:api.github.com | ソース:HighAllowCache
[2026-04-01 15:30:26] DENIED   | ビーイング:AssistantB | タイプ:FileAccess    | リソース:C:\Windows\System32 | ソース:HighDenyCache
[2026-04-01 15:30:27] ASK_USER | ビーイング:Curator    | タイプ:CommandLine   | リソース:del /f /q *.log | ソース:Callback
[2026-04-01 15:30:28] ALLOWED  | ビーイング:Curator    | タイプ:CommandLine   | リソース:del /f /q *.log | ソース:UserDecision
```

ログはストレージに永続化され、Web UI(ログコントローラー)を介して表示可能。

## トークン使用監査

`TokenUsageAuditManager`はAIトークン消費のセキュリティ関連追跡を提供:

- **リクエストごとのレコード** — すべてのAI呼び出しは、ビーイングID、モデル、プロンプトトークン、コンプリートトークン、タイムスタンプをログに記録。
- **異常検出** — 異常なトークン消費パターンは、プロンプトインジェクションまたはリソース乱用を示す可能性。
- **キュレーターのみアクセス** — `TokenAuditTool`(`[SiliconManagerOnly]`マーク付き)により、キュレーターはトークン使用をクエリおよび要約可能。
- **Webダッシュボード** — `AuditController`は、トレンドチャートとデータエクスポート付きのブラウザベースダッシュボードを提供。
- **永続ストレージ** — レコードは`ITimeStorage`を介して保存され、時系列クエリと長期分析が可能。
