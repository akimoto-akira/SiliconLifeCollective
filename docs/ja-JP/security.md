# セキュリティ設計

[English](../en/security.md) | [中文](../zh-CN/security.md) | [繁體中文](../zh-HK/security.md) | [Español](../es-ES/security.md) | **日本語** | [한국어](../ko-KR/security.md) | [Čeština](../cs-CZ/security.md)

## 概要

Silicon Life Collective のセキュリティは**多層防御**モデルに基づいています。コア原則：**すべての I/O 操作は実行器を経由する必要があり**、実行器は実行前に権限チェックを強制します。

```
ツール呼び出し → 実行器 → 権限マネージャー → 高拒否キャッシュ → 高許可キャッシュ → コールバック → ユーザーに確認
```

---

## 権限モデル

### 権限タイプ

| タイプ | 説明 |
|------|-------------|
| `NetworkAccess` | 送信 HTTP/HTTPS リクエスト |
| `CommandLine` | シェルコマンド実行 |
| `FileAccess` | ファイルおよびディレクトリ操作 |
| `Function` | 機密関数呼び出し |
| `DataAccess` | システムまたはユーザーデータへのアクセス |

### 権限結果

各権限チェックは3つの結果のいずれかを返します：

| 結果 | 動作 |
|--------|----------|
| [Deutsch](../de-DE/security.md) | **Allowed（許可）** | 操作は直ちに進行 |
| [Deutsch](../de-DE/security.md) | **Denied（拒否）** | 操作はブロックされ、監査ログに記録 |
| [Deutsch](../de-DE/security.md) | **AskUser（ユーザーに確認）** | 操作は一時停止。ユーザー確認が必要 |

### 特別なロール：シリコン管理人

シリコン管理人は最高権限レベル（`IsCurator = true`）を持つ。管理人の権限チェックは**許可**にショートカット。ユーザーが明示的にオーバーライドしない限り。

### プライベート権限マネージャー

各シリコン生命体は独自の**プライベート PermissionManager** インスタンスを持つ。権限状態は生命体間で共有されない。

## 権限検証フロー

クエリ優先度：**1. ユーザー高拒否 → 2. ユーザー高許可 → 3. コールバック関数**

```
┌─────────────┐
│ ツール       │
│ 呼び出し     │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  実行器      │────▶│ プライベート         │
│（ディスク/   │     │ 権限マネージャー     │
│ ネット/     │     │（生命体ごと）        │
│ コマンド...）│     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. IsCurator?   │──はい──▶ 許可
                  └────────┬────────┘
                           │ いいえ
                           ▼
                  ┌─────────────────┐
                  │ 2. ユーザー      │──マッチ──▶ 拒否
                  │  高拒否         │
                  │（メモリキャッシュ）│
                  └────────┬────────┘
                           │ 非マッチ
                           ▼
                  ┌─────────────────┐
                  │ 3. ユーザー      │──マッチ──▶ 許可
                  │  高許可         │
                  │（メモリキャッシュ）│
                  └────────┬────────┘
                           │ 非マッチ
                           ▼
                  ┌─────────────────┐
                  │ 4. 権限         │
                  │  コールバック    │──▶ 許可 / 拒否 / ユーザーに確認
                  └─────────────────┘
```

**重要ポイント**：実行器はブール値（許可/拒否）のみを見る。権限マネージャーは内部的に三者択一の決定（許可/拒否/ユーザーに確認）を処理し、実行器に返す前に「ユーザーに確認」を解決。

## 実行器（セキュリティ境界）

実行器は I/O 操作の**唯一**のパス。以下を強制：

### 専用ディスパッチスレッド

各実行器は**専用ディスパッチスレッド**を持つ：

- 実行器間のスレッド分離 — 1つの実行器のスレッドブロッキングが他の実行器に影響しない。
- 各実行器は独立したリソース制限（CPU、メモリなど）を設定可能。
- 実行器スレッドのスレッドプール管理。

### リクエストキュー

各実行器はリクエストキューを維持：

- リクエストはタイプに応じて対応する実行器にルーティング。
- 優先度付きキューイングをサポート。
- リクエストごとのタイムアウト制御。

### 権限検証のスレッドロック

ツールがリソースアクセスを開始した場合：

1. 実行器はリクエストを受け取り、**スレッドをロック**。
2. 実行器は生命体のプライベート権限マネージャーにクエリ。
3. コールバックが「ユーザーに確認」を返した場合、実行器スレッドはユーザー応答を待つために**ロックを保持**。
4. 生命体は最終結果（成功または拒否）のみを見る — 中間の「保留中」や「待機中」状態を見ることは決してない。
5. シリコン管理人のみが真のユーザープロンプトをトリガー。通常の生命体はブロッキングせずにグローバル ACL を同期的にクエリ。
6. タイムアウト時、リクエストは拒否として扱われ、スレッドロックは解放。

### 実行器タイプ

| 実行器 | 範囲 | デフォルトタイムアウト |
|----------|-------|-----------------|
| `DiskExecutor` | ファイル読み/書き、ディレクトリ操作 | 30秒 |
| `NetworkExecutor` | HTTP リクエスト、WebSocket 接続 | 60秒 |
| `CommandLineExecutor` | シェルコマンド実行 | 120秒 |
| `DynamicCompilationExecutor` | Roslyn メモリコンパイル | 60秒 |

### 例外分離とフォールトトレランス

- 1つの実行器の例外が他の実行器に影響しない。
- スレッドがクラッシュしたときに自動再起動。
- サーキットブレーカー：連続失敗後、実行器を一時的に停止してカスケード障害を防止。

## グローバル ACL（アクセスコントロールリスト）

ストレージに永続化された共有ルールテーブル。シリコン管理人のみが管理：

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- ルールは順番に評価。最初のマッチが勝利。
- シリコン管理人のみがグローバル ACL を変更可能（専用ツール経由）。
- 変更は直ちに有効。
- グローバル ACL は上記の各クエリの優先チェーン**内にはない** — コールバック関数によって内部的に参照。

## ユーザー周波数キャッシュ

繰り返し権限プロンプトを減らすため、システムは2つの**生命体ごと、メモリのみ**のキャッシュを維持：

| キャッシュ | 用途 |
|-------|---------|
| [Deutsch](../de-DE/security.md) | **HighAllow（高許可）** | ユーザーが頻繁に許可するリソース |
| [Deutsch](../de-DE/security.md) | **HighDeny（高拒否）** | ユーザーが頻繁に拒否するリソース |

### 動作原理

- **ユーザー選択、自動検出ではない**：「ユーザーに確認」がトリガーされたとき、ユーザーはリソースをキャッシュに追加するかどうかを選択。
- **プレフィックスマッチ**：リソースパスプレフィックスマッチをサポート（例：`network:api.example.com/*`）。
- **優先度**：高拒否是高許可よりも優先度が高い。
- **メモリのみ**：キャッシュは永続化されない。再起動時に失われる。
- **設定可能な有効期限**：ユーザーはキャッシュエントリの有効期限を設定可能。

### キャッシュ更新フロー

1. 権限コールバックが `AskUser` を返す。
2. 権限システムはカードシステム（Web UI または IM）にクエリを送信。
3. ユーザーは決定（許可/拒否）を下し、**キャッシュするかどうかを選択**。
4. カードシステムは決定 + キャッシュフラグを返す。
5. 権限システムは対応するキャッシュリストを更新。
6. 将来、キャッシュプレフィックスにマッチするリクエストは直ちに解決。

## ユーザー確認メカニズム

権限チェックが `AskUser` を返した場合：

### Web UI：インタラクティブカード

Web フロントエンドは**インタラクティブカード**を直ちに表示。内容：

- リソースタイプとパス
- 操作の説明
- 許可 / 拒否ボタン
- オプションの「常に許可」/「常に拒否」チェックボックス（周波数キャッシュに追加）

### IM（カード非対応）：ランダムコード

インタラクティブカードをサポートしないメッセージングプラットフォームの場合：

1. システムは2つのランダム6桁コードを生成：**許可コード**と**拒否コード**。
2. リソース情報と2つのコードを含むメッセージを送信。
3. ユーザーは正確な許可コードを返信して承認する必要がある。その他の返信は拒否として扱われる。
4. コードはワンタイムユース。リプレイ攻撃を防止。

### タイムアウト

- すべての「ユーザーに確認」リクエストにタイムアウトを設定。
- タイムアウト時、リクエストは**拒否**として扱われ、実行器スレッドロックは解放。

## 動的コンパイルセキュリティ

自己進化（クラスのオーバーライド）は固有のセキュリティリスクをもたらす。システムは**多層戦略**でそれらを緩和：

### レイヤー 1：コンパイル時参照制御（主要防御）

- コンパイラは**許可されたアセンブリ参照リスト**のみを取得。
- **許可**：`System.Runtime`、`System.Private.CoreLib`、プロジェクトアセンブリ（ITool インターフェースなど）
- **ブロック**：`System.IO`、`System.Reflection`、`System.Runtime.InteropServices` など
- コードがブロックされたアセンブリを参照した場合、**コンパイラ自体がコードを拒否**。
- これは実行時スキャンよりも信頼性が高い — 危険な操作はタイプレベルで不可能。

### レイヤー 2：実行時静的分析（二次防御）

- コンパイル成功後でも、コードは静的パターンスキャンを受ける。
- 危険な操作パターン（直接 I/O、システム呼び出しなど）を検出。
- 危険なコードが見つかった場合、ロードは拒否され、システムはデフォルト機能にフォールバック。

### 継承制約

すべてのカスタムシリコン生命体クラスは**必ず** `SiliconBeingBase` を継承**しなければならない**。コンパイラはタイプレベルでこの制約を強制。

### 暗号化ストレージ

コンパイルされたコードは AES-256 でディスク上に暗号化保存：

- **キー派生**：生命体の GUID（大文字）から PBKDF2 を使用。
- **復号失敗**：デフォルト実装にフォールバック。
- **実行時再コンパイル**：新しいコードはまずメモリ内でコンパイル。コンパイルとインスタンスの交換が成功した場合のみ永続化。

### アトミック置換

置換プロセスはアトミック：

1. メモリ内で新しいコードをコンパイル → `Type` を取得。
2. `Type` から新しいインスタンスを作成。
3. 古いインスタンスから新しいインスタンスに状態を移行。
4. 参照を交換。
5. 暗号化コードを永続化。

いずれかのステップが失敗した場合、古いインスタンスは生きたまま。

## 権限コールバック関数

### 設計

各 PermissionManager は**コールバック関数変数**を持つ：

- **デフォルト**：組み込みデフォルト権限関数を指す。
- **動的コンパイル後**：生命体のカスタム権限関数でオーバーライド。
- **二者択一**：いつでもアクティブなコールバックは1つのみ。
- **コンパイル失敗**：現在のコールバックに影響しない — デフォルトまたは前回の成功したカスタム関数が有効なまま。

### コールバックシグネチャ

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

`Allowed`、`Denied`、または `AskUser` を返す。

## 監査ログ

すべての権限決定が記録される：

```
[2026-04-01 15:30:25] ALLOWED  | Being:AssistantA | Type:NetworkAccess | Resource:api.github.com | Source:HighAllowCache
[2026-04-01 15:30:26] DENIED   | Being:AssistantB | Type:FileAccess    | Resource:C:\Windows\System32 | Source:HighDenyCache
[2026-04-01 15:30:27] ASK_USER | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:Callback
[2026-04-01 15:30:28] ALLOWED  | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:UserDecision
```

ログはストレージに永続化。Web UI（ログコントローラー）を介して表示可能。

## トークン使用監査

`TokenUsageAuditManager` はセキュリティ関連の AI トークン消費追跡を提供：

- **リクエストごとの記録** — 各 AI 呼び出しは生命体 ID、モデル、プロンプトートークン、補完トークン、タイムスタンプを記録。
- **異常検出** — 異常なトークン消費パターンは、プロンプト注入やリソース乱用を示す可能性がある。
- **管理人のみアクセス** — `TokenAuditTool`（`[SiliconManagerOnly]` でマーク）は、管理人がトークン使用をクエリおよび集計することを許可。
- **Web ダッシュボード** — `AuditController` は、トレンドグラフとデータエクスポート付きのブラウザベースダッシュボードを提供。
- **永続化ストレージ** — 記録は `ITimeStorage` を介して保存。時系列クエリと長期分析用。
