# セキュリティ設計

[English](security.md) | [简体中文](docs/zh-CN/security.md) | [繁體中文](docs/zh-HK/security.md) | [Español](docs/es-ES/security.md) | [日本語](docs/ja-JP/security.md) | [한국어](docs/ko-KR/security.md) | [Čeština](docs/cs-CZ/security.md)

## 概要

Silicon Life Collective のセキュリティは**多層防御**モデルに基づいています。核心原則：**すべての I/O 操作は必ずエグゼキューターを通過する必要があり**、エグゼキューターは実行前に権限チェックを強制します。

```
ツールコール → エグゼキューター → パーミッションマネージャー → HighDeny キャッシュ → HighAllow キャッシュ → コールバック → ユーザーに確認
```

---

## 権限モデル

### 権限タイプ

| タイプ | 説明 |
|------|-------------|
| `NetworkAccess` | アウトバウンド HTTP/HTTPS リクエスト |
| `CommandLine` | シェルコマンド実行 |
| `FileAccess` | ファイルおよびディレクトリ操作 |
| `Function` | 機密関数呼び出し |
| `DataAccess` | システムまたはユーザーデータへのアクセス |

### 権限結果

すべての権限チェックは 3 つの結果のいずれかを返します：

| 結果 | 動作 |
|--------|----------|
| **Allowed** | 操作は即座に進行 |
| **Denied** | 操作はブロックされ、監査ログが記録 |
| **AskUser** | 操作は一時停止され、ユーザー確認が必要 |

### 特別な役割：シリコンキュレーター

シリコンキュレーターは最高の権限レベル（`IsCurator = true`）を保持します。キュレーターの権限チェックは、ユーザーによって明示的にオーバーライドされない限り、**Allowed** にショートサーキットされます。

### プライベート権限マネージャー

各シリコンビーイングは独自の**プライベート PermissionManager** インスタンスを持ちます。権限状態はビーイング間で共有されません。

## 権限検証フロー

クエリ優先度は：**1. ユーザー HighDeny → 2. ユーザー HighAllow → 3. コールバック関数**

```
┌─────────────┐
│ ツールコール  │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  エグゼキューター │────▶│ プライベート権限      │
│ （Disk/Net/  │     │ マネージャー（生命体ごと） │
│  CLI/...）    │     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. IsCurator?   │──Yes──▶ Allowed
                  └────────┬────────┘
                           │ No
                           ▼
                  ┌─────────────────┐
                  │ 2. ユーザー HighDeny│──Match──▶ Denied
                  │ （メモリキャッシュ）  │
                  └────────┬────────┘
                           │ Miss
                           ▼
                  ┌─────────────────┐
                  │ 3. ユーザー HighAllow│──Match──▶ Allowed
                  │ （メモリキャッシュ）  │
                  └────────┬────────┘
                           │ Miss
                           ▼
                  ┌─────────────────┐
                  │ 4. 権限          │
                  │  コールバック関数  │──▶ Allowed / Denied / AskUser
                  └─────────────────┘
```

**重要ポイント**：エグゼキューターはブール値（Allowed/Denied）のみを見ます。パーミッションマネージャーは内部的に三者決定（Allowed/Denied/AskUser）を処理し、エグゼキューターに戻る前に AskUser を解決します。

## エグゼキューター（セキュリティ境界）

エグゼキューターは I/O 操作の**唯一の**パスです。以下を強制します：

### 独立したスケジューリングスレッド

各エグゼキューターは**独立したスケジューリングスレッド**を所有します：

- スレッドはエグゼキューター間で分離されます — 1 つのエグゼキュータースレッドのブロックは他のエグゼキューターに影響を与えません。
- 各エグゼキューターは独立したリソース制限（CPU、メモリなど）を設定できます。
- エグゼキュータースレッドのスレッドプール管理。

### リクエストキュー

各エグゼキューターはリクエストキューを維持します：

- リクエストはタイプごとに対応するエグゼキューターにルーティングされます。
- 優先度キューイングをサポート。
- リクエストごとのタイムアウト制御。

### 権限検証のためのスレッドロック

ツールがリソースアクセスを開始した場合：

1. エグゼキューターはリクエストを受信し、**スレッドをロック**します。
2. エグゼキューターは生命体のプライベート PermissionManager にクエリを実行します。
3. コールバックが AskUser を返した場合、エグゼキュータースレッドはユーザーの応答を待って**ロックされたまま**になります。
4. 生命体は最終結果（成功または拒否）のみを表示します — 中間の「保留中」または「待機中」状態は表示されません。
5. シリコンキュレーターのみが実際のユーザープロンプトをトリガーします。通常の生命体はブロックせずにグローバル ACL を同期的にクエリします。
6. タイムアウト時、リクエストは拒否され、スレッドロックが解放されます。

### エグゼキュータータイプ

| エグゼキューター | 範囲 | デフォルトタイムアウト |
|----------|-------|-----------------|
| `DiskExecutor` | ファイル読み取り/書き込み、ディレクトリ操作 | 30 秒 |
| `NetworkExecutor` | HTTP リクエスト、WebSocket 接続 | 60 秒 |
| `CommandLineExecutor` | シェルコマンド実行 | 120 秒 |
| `DynamicCompilationExecutor` | Roslyn メモリ内コンパイル | 60 秒 |

### 例外分離とフォールトトレランス

- 1 つのエグゼキューターの例外は他のエグゼキューターに影響を与えません。
- スレッドクラッシュ時の自動再起動。
- サーキットブレーカー：連続した失敗後、エグゼキューターを一時的に停止してカスケード障害を防止。

## グローバル ACL（アクセスコントロールリスト）

シリコンキュレーターによって独占的に管理される、ストレージに永続化された共有ルールテーブル：

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- ルールは順番に評価されます；最初の一致が優先されます。
- シリコンキュレーターのみがグローバル ACL を変更できます（専用ツールを介して）。
- 変更は即座に反映されます。
- グローバル ACL は上記のクエリ優先度チェーンには**ありません** — コールバック関数によって内部的に参照されます。

## ユーザー頻度キャッシュ

繰り返し権限プロンプトを減らすために、システムは**生命体ごと、メモリのみ**の 2 つのキャッシュを維持します：

| キャッシュ | 目的 |
|-------|---------|
| **HighAllow** | ユーザーが頻繁に許可したリソース |
| **HighDeny** | ユーザーが頻繁に拒否したリソース |

### 動作原理

- **ユーザーオプトイン、自動検出ではない**：AskUser がトリガーされた場合、ユーザーはリソースをキャッシュに追加するかどうかを選択します。
- **プレフィックスマッチング**：リソースパスプレフィックスマッチングをサポート（例：`network:api.example.com/*`）。
- **優先度**：HighDeny は HighAllow より高い優先度を持ちます。
- **メモリのみ**：キャッシュは永続化されません。再起動時に失われます。
- **有効期限設定可能**：ユーザーはキャッシュエントリの有効期間を設定できます。

### キャッシュ更新フロー

1. 権限コールバックが `AskUser` を返します。
2. 権限システムはカードシステム（Web UI または IM）に問い合わせを送信します。
3. ユーザーは決定（許可/拒否）を行い、**キャッシュするかどうかを選択**します。
4. カードシステムは決定 + キャッシュフラグを返します。
5. 権限システムは対応するキャッシュリストを更新します。
6. キャッシュされたプレフィックスに一致する将来のリクエストは即座に解決されます。

## ユーザー確認メカニズム

権限チェックが `AskUser` を返した場合：

### Web UI：インタラクティブカード

Web フロントエンドは以下を示す**インタラクティブカード**を即座に表示します：

- リソースタイプとパス
- 操作の説明
- 許可 / 拒否 ボタン
- オプションの「常に許可」/「常に拒否」チェックボックス（頻度キャッシュに追加）

### IM（カードサポートなし）：ランダムコード

インタラクティブカードをサポートしないメッセージングプラットフォームの場合：

1. システムは 2 つのランダムな 6 桁コードを生成します：**許可コード**と**拒否コード**。
2. リソース情報と両方のコードを含むメッセージを送信します。
3. ユーザーは正確な許可コードを返信して承認する必要があります。他の返信は拒否として扱われます。
4. コードはワンタイム使用で、リプレイ攻撃を防止します。

### タイムアウト

- すべての AskUser リクエストにタイムアウトが設定されます。
- タイムアウト時、リクエストは**拒否**され、エグゼキュータースレッドロックが解放されます。

## 動的コンパイルセキュリティ

自己進化（クラス書き換え）は独自のセキュリティリスクをもたらします。システムは**多層戦略**でこれらを軽減します：

### 層 1：コンパイル時参照制御（主要防御）

- コンパイラには**許可されたアセンブリ参照リスト**のみが提供されます。
- **許可**：`System.Runtime`、`System.Private.CoreLib`、プロジェクトアセンブリ（ITool インターフェースなど）
- **ブロック**：`System.IO`、`System.Reflection`、`System.Runtime.InteropServices` など
- コードがブロックされたアセンブリを参照している場合、**コンパイラ自体がコードを拒否**します。
- これは実行時スキャンよりも信頼性が高い — 危険な操作は型レベルで不可能です。

### 層 2：実行時静的分析（二次防御）

- コンパイル成功後でも、コードは静的パターンスキャンを受けます。
- 危険な操作パターン（直接 I/O、システムコールなど）を検出。
- 危険なコードが見つかった場合、読み込みが拒否され、システムはデフォルト関数にフォールバックします。

### 継承制約

すべてのカスタムシリコンビーイングクラスは**必ず** `SiliconBeingBase` を継承する必要があります。コンパイラはこれを型レベルで強制します。

### 暗号化ストレージ

コンパイルされたコードはディスク上で AES-256 暗号化で保存されます：

- **キー派生**：生命体の GUID（大文字）からの PBKDF2。
- **復号化失敗**：デフォルト実装にフォールバック。
- **実行時再コンパイル**：新しいコードはまずメモリ内でコンパイルされます；コンパイル成功とインスタンス置換後にのみ永続化されます。

### アトミック置換

置換プロセスはアトミックです：

1. メモリ内で新しいコードをコンパイル → `Type` を取得。
2. `Type` から新しいインスタンスを作成。
3. 古いインスタンスから新しいインスタンスに状態を移行。
4. 参照をスワップ。
5. 暗号化されたコードを永続化。

いずれかのステップが失敗した場合、古いインスタンスがアクティブのままになります。

## 権限コールバック関数

### 設計

各 PermissionManager は**コールバック関数変数**を保持します：

- **デフォルト**：組み込みデフォルト権限関数を指します。
- **動的コンパイル後**：生命体のカスタム権限関数によってオーバーライドされます。
- **二者択一**：いつでもアクティブなのは 1 つのコールバックのみ。
- **コンパイル失敗**：現在のコールバックに影響を与えません — デフォルトまたは最後の成功したカスタム関数が有効のままになります。

### コールバックシグネチャ

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

`Allowed`、`Denied`、または `AskUser` を返します。

## 監査ログ

すべての権限決定がログに記録されます：

```
[2026-04-01 15:30:25] ALLOWED  | Being:AssistantA | Type:NetworkAccess | Resource:api.github.com | Source:HighAllowCache
[2026-04-01 15:30:26] DENIED   | Being:AssistantB | Type:FileAccess    | Resource:C:\Windows\System32 | Source:HighDenyCache
[2026-04-01 15:30:27] ASK_USER | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:Callback
[2026-04-01 15:30:28] ALLOWED  | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:UserDecision
```

ログはストレージに永続化され、Web UI（Log コントローラー）を介して表示できます。

## トークン使用監査

`TokenUsageAuditManager` は AI トークン消費のセキュリティ関連追跡を提供します：

- **リクエストごとのレコード** — すべての AI コールは生命体 ID、モデル、プロンプトトークン、完了トークン、タイムスタンプをログに記録。
- **異常検出** — 通常と異なるトークン消費パターンは、プロンプトインジェクションまたはリソース乱用を示す可能性があります。
- **キュレーター専用アクセス** — `TokenAuditTool`（`[SiliconManagerOnly]` マーク付き）により、キュレーターはトークン使用をクエリおよび要約できます。
- **Web ダッシュボード** — `AuditController` はトレンドチャートとデータエクスポート付きのブラウザベースダッシュボードを提供。
- **永続ストレージ** — レコードは時系列クエリと長期分析のために `ITimeStorage` を介して保存されます。
