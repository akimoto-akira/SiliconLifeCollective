## コアコンセプト

### シリコン生命体

システムの各 AI エージェントは**シリコン生命体**です — 独自のアイデンティティ、個性、能力を持つ自律エンティティ。各シリコン生命体は**ソウルファイル**（Markdown プロンプト）によって駆動され、行動パターンを定義します。

### シリコン管理人

**シリコン管理人**は最高システム権限を持つ特別なシリコン生命体です。システム管理者として機能：

- 他のシリコン生命体の作成と管理
- ユーザーリクエストの分析とタスクへの分解
- 適切なシリコン生命体へのタスク配布
- 実行品質のモニタリングと失敗の処理
- **優先スケジューリング**でユーザーメッセージに応答（下記参照）

### ソウルファイル

各シリコン生命体のデータディレクトリに保存される Markdown ファイル（`soul.md`）。システムプロンプトとして各 AI リクエストに注入され、生命体の個性、決定パターン、行動制約を定義します。

## スケジューリング：タイムスロット公平スケジューリング

### メインループ + クロックオブジェクト

システムは専用バックグラウンドスレッドで**クロック駆動のメインループ**を実行：

```
メインループ（専用スレッド、ウォッチドッグ + サーキットブレーカー）
  └── クロックオブジェクト A（優先度=0、間隔=100ms）
  └── クロックオブジェクト B（優先度=1、間隔=500ms）
  └── シリコン生命体マネージャー（メインループから直接クロックトリガー）
        └── シリコン生命体ランナー → 生命体 1 → クロックトリガー → 1ラウンド実行
        └── シリコン生命体ランナー → 生命体 2 → クロックトリガー → 1ラウンド実行
        └── シリコン生命体ランナー → 生命体 3 → クロックトリガー → 1ラウンド実行
        └── ...
```

重要な設計決定：

- **シリコン生命体はクロックオブジェクトを継承しません。** 独自の `Tick()` メソッドを持ち、`SiliconBeingManager` が `SiliconBeingRunner` を介して呼び出します。メインループに直接登録されません。
- **シリコン生命体マネージャー**はメインループから直接クロックトリガーされ、すべての生命体の単一エージェントとして機能。
- **シリコン生命体ランナー**は各生命体の `Tick()` を一時スレッドでラップ。タイムアウトと生命体ごとのサーキットブレーカー（3回連続タイムアウト → 1分クールダウン）付き。
- 各生命体の実行は、クロックトリガーごとに**1ラウンド**の AI リクエスト + ツール呼び出しに制限。生命体がメインループを独占できないことを保証。
- **パフォーマンスモニタ**は観測可能性のためにクロック実行時間を追跡。

### 管理人優先応答

ユーザーがシリコン管理人にメッセージを送信した場合：

1. 現在の生命体（例：生命体 A）が現在のラウンドを完了 — **中断しない**。
2. マネージャーが**残りのキューをスキップ**。
3. ループが**管理人から再開**され、即座に実行。

これにより、進行中のタスクを妨害せずにユーザーインタラクションに応答することを保証。

## チャットシステム

### セッションタイプ

チャットシステムは `SessionBase` を介して 3 種類のセッションタイプをサポート：

| タイプ | クラス | 説明 |
|------|-------|-------------|
| `SingleChat` | `SingleChatSession` | 2人の参加者間の1対1の会話 |
| `GroupChat` | `GroupChatSession` | 複数参加者のグループチャット |
| `Broadcast` | `BroadcastChannel` | 固定 ID のオープンチャンネル。生命体は動的にサブスクライブ。サブスクライブ後のみメッセージを受信 |

### ブロードキャストチャンネル

`BroadcastChannel` はシステム全体のアナウンス用の特別なセッションタイプ：

- **固定チャンネル ID** — `SingleChatSession` や `GroupChatSession` と異なり、チャンネル ID はメンバー GUID から派生しない周知の定数。
- **動的サブスクリプション** — 生命体は実行時にサブスクライブ/アンサブスクライブ。サブスクライブ後に投稿されたメッセージのみを受信。
- **保留メッセージフィルタリング** — `GetPendingMessages()` は生命体のサブスクライブ時間後に投稿され、未読のメッセージのみを返す。
- **チャットシステムによる管理** — `GetOrCreateBroadcastChannel()`、`Broadcast()`、`GetPendingBroadcasts()`。

### チャットメッセージ

`ChatMessage` モデルには、AI 会話コンテキストとトークン追跡のフィールドが含まれる：

| フィールド | タイプ | 説明 |
|-------|------|-------------|
| `Id` | `Guid` | 一意のメッセージ識別子 |
| `SenderId` | `Guid` | 送信者の一意の識別子 |
| `ChannelId` | `Guid` | チャンネル/会話識別子 |
| `Content` | `string` | メッセージコンテンツ |
| `Timestamp` | `DateTime` | メッセージ送信時間 |
| `Type` | `MessageType` | テキスト、画像、ファイル、またはシステム通知 |
| `ReadBy` | `List<Guid>` | このメッセージを読んだ参加者 ID |
| `Role` | `MessageRole` | AI 会話ロール（ユーザー、アシスタント、ツール） |
| `ToolCallId` | `string?` | ツール結果メッセージのツール呼び出し ID |
| `ToolCallsJson` | `string?` | アシスタントメッセージのシリアライズされたツール呼び出し JSON |
| `Thinking` | `string?` | AI の思考連鎖推論 |
| `PromptTokens` | `int?` | プロンプトのトークン数（入力） |
| `CompletionTokens` | `int?` | 補完のトークン数（出力） |
| `TotalTokens` | `int?` | 使用された総トークン数（入力 + 出力） |
| `FileMetadata` | `FileMetadata?` | 添付ファイルメタデータ（メッセージにファイルが含まれる場合） |

### チャットメッセージキュー

`ChatMessageQueue` は、チャットメッセージの非同期処理を管理するスレッドセーフなメッセージキューシステム：

- **スレッドセーフ** - ロックメカニズムを使用して安全な並行アクセスを保証
- **非同期処理** - 非同期メッセージエンキューとデキューをサポート
- **メッセージソート** - メッセージの時間順序を維持
- **バッチ操作** - バッチメッセージ取得をサポート

### ファイルメタデータ

`FileMetadata` は、チャットメッセージに添付されたファイル情報を管理：

- **ファイル情報** - ファイル名、サイズ、タイプ、パス
- **アップロード時間** - ファイルアップロードのタイムスタンプ
- **アップローダー** - ファイルをアップロードしたユーザーまたはシリコン生命体 ID

### ストリームキャンセルマネージャー

`StreamCancellationManager` は AI ストリーミング応答のキャンセルメカニズムを提供：

- **ストリーム制御** - 進行中の AI ストリーミング応答のキャンセルをサポート
- **リソースクリーンアップ** - キャンセル時に関連リソースを適切にクリーンアップ
- **並行安全** - 複数のストリームの同時管理をサポート

### チャット履歴表示

新しいチャット履歴表示機能により、ユーザーはシリコン生命体の過去の会話を閲覧可能：

- **セッションリスト** - すべての過去のセッションを表示
- **メッセージ詳細** - 完全なメッセージ履歴を表示
- **タイムラインビュー** - 時間順にメッセージを表示
- **API サポート** - セッションとメッセージデータを取得するための RESTful API を提供

## 重要な設計決定

### ストレージをインスタンスクラスとして（静的ではなく）

`IStorage` は静的ユーティリティではなく、注入可能なインスタンスとして設計。これにより保証：

- 直接ファイルシステムアクセス — IStorage はシステムの内部永続化チャネル。実行器を経由**しない**。
- **AI は IStorage を制御できない** — 実行器は AI ツールによって開始された IO を管理。IStorage はフレームワーク独自の内部データ読み書きを管理。これらは根本的に異なる関心事。
- モック実装を使用したテストが可能。
- 消費者を変更せずに、将来異なるストレージバックエンドをサポート。

### 実行器をセキュリティ境界として

実行器は I/O 操作の**唯一**のパス。ディスク、ネットワーク、またはコマンドラインアクセスを必要とするツールは**必ず**実行器を経由する必要があります。この設計は強制：

- 各実行器は**専用ディスパッチスレッド**を持つ。権限検証用のスレッドロック付き。
- 集中化された権限チェック — 実行器は生命体の**プライベート権限マネージャー**にクエリ。
- 優先順位とタイムアウト制御をサポートするリクエストキュー。
- すべての外部操作の監査ログ。
- 例外分離 — 1つの実行器の失敗が他の実行器に影響しない。
- サーキットブレーカー — 連続失敗で一時的に実行器を停止し、カスケード障害を防止。

### ContextManager を軽量オブジェクトとして

各 `ExecuteOneRound()` が新しい `ContextManager` インスタンスを作成：

1. ソウルファイル + 最近のチャット履歴をロード。
2. AI クライアントにリクエストを送信。
3. AI がプレーンテキストを返すまでツール呼び出しをループ処理。
4. チャットシステムに応答を永続化。
5. 破棄。

これにより、各ラウンドを分離されたステートレスに保つ。

### クラスのオーバーライドによる自己進化

シリコン生命体は実行時に自分の C# クラスを書き換え可能：

1. AI が新しいクラスコードを生成（`SiliconBeingBase` を継承する必要がある）。
2. **コンパイル時参照制御**（主要防御）：コンパイラは許可されたアセンブリリストのみを取得 — `System.IO`、`System.Reflection` などは除外されるため、危険なコードはタイプレベルで不可能。
3. **実行時静的分析**（二次防御）：`SecurityScanner` はコンパイル成功後にコード内の危険なパターンをスキャン。
4. Roslyn がコードをメモリ内でコンパイル。
5. 成功時：`SiliconBeingManager.ReplaceBeing()` が現在のインスタンスを交換。状態を移行し、暗号化されたコードをディスクに永続化。
6. 失敗時：新しいコードを破棄。既存の実装を保持。

カスタム `IPermissionCallback` 実装も `ReplacePermissionCallback()` を介してコンパイルおよび注入可能。生命体が独自の権限ロジックをカスタマイズできる。

コードは AES-256 でディスク上に暗号化保存。暗号化キーは生命体の GUID（大文字）から PBKDF2 を介して派生。

## トークン使用監査

`TokenUsageAuditManager` はすべての生命体の AI トークン消費を追跡：

- `TokenUsageRecord` — リクエストごとの記録（生命体 ID、モデル、プロンプトートークン、補完トークン、タイムスタンプ）
- `TokenUsageSummary` — 集計統計
- `TokenUsageQuery` — 記録をフィルタするためのクエリパラメータ
- 時系列クエリのために `ITimeStorage` を介して永続化
- Web UI（AuditController）および `TokenAuditTool`（管理人のみ）を介してアクセス可能

## Web UI アーキテクチャ

### スキンシステム

Web UI には**プラグイン可能なスキンシステム**があり、アプリケーションロジックを変更せずに完全な UI カスタマイズが可能：

- **ISkin インターフェース** — すべてのスキンの契約を定義。含む：
  - コアレンダリングメソッド（`RenderHtml`、`RenderError`）
  - 20+ の UI コンポーネントメソッド（ボタン、入力、カード、テーブル、バッジ、バブル、プログレス、タグなど）
  - `CssBuilder` を介してテーマ CSS を生成
  - `SkinPreviewInfo` — 初期化ページスキンのセレクター用のカラーパレットとアイコン

- **組み込みスキン** — 4つの本番準備完了スキン：
  - **Admin** — プロフェッショナル、データフォーカスのシステム管理インターフェース
  - **Chat** — 対話型、メッセージ中心のデザイン。AI インタラクション用
  - **Creative** — アーティスティック、視覚的に豊富なクリエイティブワークフローレイアウト
  - **Dev** — 開発者中心、コード中心のインターフェース。シンタックスハイライト付き

- **スキン発見** — `SkinManager` はリフレクションを介してすべての `ISkin` 実装を自動的に発見および登録

### HTML / CSS / JS ビルダー

Web UI はテンプレートファイルを完全に回避。C# ですべてのマークアップを生成：

- **`H`** — ストリーミング HTML ビルダー DSL。コード内で HTML ツリーを構築
- **`CssBuilder`** — CSS ビルダー。セレクターとメディアクエリをサポート
- **`JsBuilder`（`JsSyntax`）** — JavaScript ビルダー。インラインスクリプト用

- **WebSocket は不要** — SSE を使用してほとんどのリアルタイムニーズに対応するよりシンプルなアーキテクチャ
- **自動再接続** — クライアント再接続ロジックによる弾力性のある接続

### WebView ブラウザ自動化システム（新規）

システムは**Playwright** ベースの WebView ブラウザ自動化機能を統合：

- **個別分離**：各シリコン生命体は独自のブラウザインスタンス、Cookie、セッションストレージを持つ。完全に分離され、干渉しない。
- **ヘッドレスモード**：ブラウザはユーザーが完全に不可視のヘッドレスモードで実行。シリコン生命体はバックグラウンドで自律的に操作。
- **WebViewBrowserTool**：完全なブラウザ操作機能を提供。含む：
  - ページナビゲーション、クリック、テキスト入力、ページコンテンツ取得
  - JavaScript 実行、スクリーンショット取得、要素の出現を待機
  - ブラウザ状態管理とリソースクリーンアップ
- **セキュリティ制御**：すべてのブラウザ操作は権限検証チェーンを通過する必要があり、悪意のある Web ページアクセスを防止。

### ナレッジネットワークシステム（新規）

システムは**三つ組構造**に基づくナレッジグラフシステムを内蔵：

- **ナレッジ表現**：「主語-関係-目的語」の三つ組構造を採用（例：Python-is_a-programming_language）
- **KnowledgeTool**：ナレッジの全ライフサイクル管理を提供：
  - `add`/`query`/`update`/`delete` - 基本 CRUD 操作
  - `search` - フルテキスト検索とキーワードマッチング
  - `get_path` - 2つの概念間の関連パスを発見
  - `validate` - ナレッジ完全性チェック
  - `stats` - ナレッジネットワーク統計分析
- **永続化ストレージ**：ナレッジ三つ組はファイルシステムに永続化。時間インデックスクエリをサポート。
- **信頼度スコア**：各ナレッジエントリには信頼度スコア（0-1）が付与。ナレッジのあいまいマッチングとソートをサポート。
- **タグ分類**：ナレッジにタグを追加可能。分類と検索を容易にする。
