# 安全设计

> **版本：v0.1.0-alpha**

[English](../en/security.md) | [Deutsch](../de-DE/security.md) | **中文** | [繁體中文](../zh-HK/security.md) | [Español](../es-ES/security.md) | [日本語](../ja-JP/security.md) | [한국어](../ko-KR/security.md) | [Čeština](../cs-CZ/security.md)

## 概述

Silicon Life Collective 的安全性建立在**分层防御**模型之上。核心原则：**所有 I/O 操作必须通过执行器**，执行器在执行前强制执行权限检查。

```
工具调用 → 执行器 → 权限管理器 → 高拒绝缓存 → 高允许缓存 → 回调 → 询问用户
```

---

## 权限模型

### 权限类型

| 类型 | 描述 |
|------|-------------|
| `NetworkAccess` | 出站 HTTP/HTTPS 请求 |
| `CommandLine` | Shell 命令执行 |
| `FileAccess` | 文件和目录操作 |
| `Function` | 敏感函数调用 |
| `DataAccess` | 访问系统或用户数据 |

### 权限结果

每次权限检查返回三种结果之一：

| 结果 | 行为 |
|--------|----------|
| [Deutsch](../de-DE/security.md) | **Allowed（允许）** | 操作立即进行 |
| [Deutsch](../de-DE/security.md) | **Denied（拒绝）** | 操作被阻止，记录审计日志 |
| [Deutsch](../de-DE/security.md) | **AskUser（询问用户）** | 操作暂停，需要用户确认 |

### 特殊角色：硅基主理人

硅基主理人拥有最高权限级别（`IsCurator = true`）。主理人的权限检查被短路为**允许**，除非用户明确覆盖。

### 私有权限管理器

每个硅基生命体都有自己的**私有 PermissionManager** 实例。权限状态不在生命体之间共享。

---

## 权限验证流程

查询优先级为：**1. 用户高拒绝 → 2. 用户高允许 → 3. 回调函数**

```
┌─────────────┐
│ 工具调用     │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  执行器      │────▶│ 私有权限            │
│（磁盘/网络/  │     │ 管理器（每个生命体）  │
│  命令行...） │     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. IsCurator?   │──是──▶ 允许
                  └────────┬────────┘
                           │ 否
                           ▼
                  ┌─────────────────┐
                  │ 2. 用户高拒绝    │──匹配──▶ 拒绝
                  │（内存缓存）      │
                  └────────┬────────┘
                           │ 未匹配
                           ▼
                  ┌─────────────────┐
                  │ 3. 用户高允许    │──匹配──▶ 允许
                  │（内存缓存）      │
                  └────────┬────────┘
                           │ 未匹配
                           ▼
                  ┌─────────────────┐
                  │ 4. 权限         │
                  │  回调函数        │──▶ 允许 / 拒绝 / 询问用户
                  └─────────────────┘
```

**关键点**：执行器只看到布尔值（允许/拒绝）。权限管理器内部处理三元决策（允许/拒绝/询问用户），并在返回给执行器之前解决询问用户。

## 执行器（安全边界）

执行器是 I/O 操作的**唯一**路径。它们强制执行：

### 独立调度线程

每个执行器拥有**独立的调度线程**：

- 执行器之间的线程隔离 —— 一个执行器的线程阻塞不影响其他执行器。
- 每个执行器可以设置独立的资源限制（CPU、内存等）。
- 执行器线程的线程池管理。

### 请求队列

每个执行器维护一个请求队列：

- 请求按类型路由到相应的执行器。
- 支持优先级排队。
- 每个请求的超时控制。

### 权限验证的线程锁定

当工具发起资源访问时：

1. 执行器接收请求并**锁定其线程**。
2. 执行器查询生命体的私有权限管理器。
3. 如果回调返回询问用户，执行器线程**保持锁定**等待用户响应。
4. 生命体只看到最终结果（成功或拒绝） —— 它永远不会看到中间的"待处理"或"等待"状态。
5. 只有硅基主理人会触发真正的用户提示。普通生命体同步查询全局 ACL 而不阻塞。
6. 超时时，请求被视为拒绝，线程锁被释放。

### 执行器类型

| 执行器 | 范围 | 默认超时 |
|----------|-------|-----------------|
| `DiskExecutor` | 文件读/写、目录操作 | 30 秒 |
| `NetworkExecutor` | HTTP 请求、WebSocket 连接 | 60 秒 |
| `CommandLineExecutor` | Shell 命令执行 | 120 秒 |
| `DynamicCompilationExecutor` | Roslyn 内存编译 | 60 秒 |

### 异常隔离与容错

- 一个执行器的异常不影响其他执行器。
- 线程崩溃时自动重启。
- 熔断器：连续失败后暂时停止执行器以防止级联失败。

## 全局 ACL（访问控制列表）

共享规则表持久化到存储，仅由硅基主理人管理：

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- 规则按顺序评估；第一次匹配获胜。
- 只有硅基主理人可以修改全局 ACL（通过其专用工具）。
- 更改立即生效。
- 全局 ACL **不在**上述每个查询的优先级链中 —— 它在内部由回调函数引用。

## 用户频率缓存

为减少重复的权限提示，系统维护两个**每个生命体、仅内存**的缓存：

| 缓存 | 用途 |
|-------|---------|
| [Deutsch](../de-DE/security.md) | **HighAllow（高允许）** | 用户频繁允许的资源 |
| [Deutsch](../de-DE/security.md) | **HighDeny（高拒绝）** | 用户频繁拒绝的资源 |

### 工作原理

- **用户选择，非自动检测**：当触发询问用户时，用户选择是否将资源添加到缓存。
- **前缀匹配**：支持资源路径前缀匹配（例如 `network:api.example.com/*`）。
- **优先级**：高拒绝的优先级高于高允许。
- **仅内存**：缓存不持久化。重启时丢失。
- **可配置过期**：用户可以设置缓存条目的有效期限。

### 缓存更新流程

1. 权限回调返回 `AskUser`。
2. 权限系统向卡片系统（Web UI 或即时通讯）发送查询。
3. 用户做出决策（允许/拒绝）并**选择是否缓存**。
4. 卡片系统返回决策 + 缓存标志。
5. 权限系统更新相应的缓存列表。
6. 未来匹配缓存前缀的请求立即解决。

## 用户询问机制

当权限检查返回 `AskUser` 时：

### Web UI：交互式卡片

Web 前端立即显示**交互式卡片**，显示：

- 资源类型和路径
- 操作描述
- 允许 / 拒绝按钮
- 可选的"始终允许" / "始终拒绝"复选框（添加到频率缓存）

### 即时通讯（无卡片支持）：随机码

对于不支持交互式卡片的消息传递平台：

1. 系统生成两个随机 6 位代码：**允许代码**和**拒绝代码**。
2. 发送包含资源信息和两个代码的消息。
3. 用户必须回复准确的允许代码才能授权。任何其他回复被视为拒绝。
4. 代码是单次使用的，以防止重放攻击。

### 超时

- 为所有询问用户请求设置超时。
- 超时时，请求被视为**拒绝**，执行器线程锁被释放。

## 动态编译安全

自我进化（类重写）引入了独特的安全风险。系统使用**分层策略**缓解它们：

### 第 1 层：编译时引用控制（主要防御）

- 编译器只获得**允许的装配引用列表**。
- **允许**：`System.Runtime`、`System.Private.CoreLib`、项目装配（ITool 接口等）
- **阻止**：`System.IO`、`System.Reflection`、`System.Runtime.InteropServices` 等
- 如果代码引用了被阻止的装配，**编译器本身拒绝**代码。
- 这比运行时扫描更可靠 —— 危险操作在类型级别是不可能的。

### 第 2 层：运行时静态分析（次要防御）

- 即使成功编译后，代码也要进行静态模式扫描。
- 检测危险操作模式（直接 I/O、系统调用等）。
- 如果发现危险代码，加载被拒绝，系统回退到默认功能。

### 继承约束

所有自定义硅基生命体类**必须**继承 `SiliconBeingBase`。编译器在类型级别强制执行此约束。

### 加密存储

编译的代码在磁盘上以 AES-256 加密存储：

- **密钥派生**：从生命体的 GUID（大写）使用 PBKDF2。
- **解密失败**：回退到默认实现。
- **运行时重新编译**：新代码先在内存中编译；只有在成功编译和实例替换后才持久化。

### 原子替换

替换过程是原子的：

1. 在内存中编译新代码 → 获取 `Type`。
2. 从 `Type` 创建新实例。
3. 从旧实例迁移状态到新实例。
4. 交换引用。
5. 持久化加密代码。

如果任何步骤失败，旧实例保持活动。

## 权限回调函数

### 设计

每个 PermissionManager 持有**回调函数变量**：

- **默认**：指向内置默认权限函数。
- **动态编译后**：被生命体的自定义权限函数覆盖。
- **二选一**：任何时候只有一个回调处于活动状态。
- **编译失败**：不影响当前回调 —— 默认或上次成功的自定义函数保持有效。

### 回调签名

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

返回 `Allowed`、`Denied` 或 `AskUser`。

## 审计日志

所有权限决策都被记录：

```
[2026-04-01 15:30:25] ALLOWED  | Being:AssistantA | Type:NetworkAccess | Resource:api.github.com | Source:HighAllowCache
[2026-04-01 15:30:26] DENIED   | Being:AssistantB | Type:FileAccess    | Resource:C:\Windows\System32 | Source:HighDenyCache
[2026-04-01 15:30:27] ASK_USER | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:Callback
[2026-04-01 15:30:28] ALLOWED  | Being:Curator    | Type:CommandLine   | Resource:del /f /q *.log | Source:UserDecision
```

日志持久化到存储，可通过 Web UI（日志控制器）查看。

## Token 使用审计

`TokenUsageAuditManager` 提供与安全性相关的 AI token 消耗跟踪：

- **每次请求记录** —— 每次 AI 调用记录生命体 ID、模型、提示词 token、补全 token 和时间戳。
- **异常检测** —— 异常的 token 消耗模式可能表明提示词注入或资源滥用。
- **仅主理人访问** —— `TokenAuditTool`（标记为 `[SiliconManagerOnly]`）允许主理人查询和汇总 token 使用。
- **Web 仪表板** —— `AuditController` 提供基于浏览器的仪表板，带趋势图和数据导出。
- **持久化存储** —— 记录通过 `ITimeStorage` 存储，用于时间序列查询和长期分析。
