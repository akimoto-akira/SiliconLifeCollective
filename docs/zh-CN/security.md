# 安全设计

[English](../en-US/security.md)

## 概述

硅基生命群的安全体系基于**分层防御**模型。核心原则：**所有 I/O 操作必须通过执行器**，执行器在执行前强制进行权限验证。

```
工具调用 → 执行器 → 权限管理器 → 高频拒绝缓存 → 高频允许缓存 → 回调函数 → 询问用户
```

---

## 权限模型

### 权限类型

| 类型 | 说明 |
|------|------|
| `NetworkAccess` | 出站 HTTP/HTTPS 请求 |
| `CommandLine` | Shell 命令执行 |
| `FileAccess` | 文件和目录操作 |
| `Function` | 敏感函数调用 |
| `DataAccess` | 系统或用户数据访问 |

### 权限判定结果

每次权限检查返回三种结果之一：

| 结果 | 行为 |
|------|------|
| **Allowed** | 操作立即执行 |
| **Denied** | 操作被阻止，记录审计日志 |
| **AskUser** | 操作暂停，等待用户确认 |

### 特殊角色：硅基主理人

硅基主理人拥有最高权限级别（`IsCurator = true`）。主理人的权限检查会被短路为 **Allowed**，除非用户显式覆盖。

### 私有权限管理器

每个硅基人拥有自己独立的**私有权限管理器**实例。权限状态不在硅基人之间共享。

---

## 权限验证流程

查询优先级：**1. 用户高频拒绝 → 2. 用户高频允许 → 3. 权限回调函数**

```
┌─────────────┐
│ 工具调用     │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────────────┐
│  执行器     │────▶│ 私有权限管理器      │
│（磁盘/网络/ │     │（每个硅基人独立）   │
│ 命令行/...）│     └────────┬────────────┘
└─────────────┘            │
                           ▼
                  ┌─────────────────┐
                  │ 1. 是主理人？   │──是──▶ Allowed
                  └────────┬────────┘
                           │ 否
                           ▼
                  ┌─────────────────┐
                  │ 2. 用户高频拒绝 │──命中──▶ Denied
                  │（内存缓存）     │
                  └────────┬────────┘
                           │ 未命中
                           ▼
                  ┌─────────────────┐
                  │ 3. 用户高频允许 │──命中──▶ Allowed
                  │（内存缓存）     │
                  └────────┬────────┘
                           │ 未命中
                           ▼
                  ┌─────────────────┐
                  │ 4. 权限         │
                  │  回调函数       │──▶ Allowed / Denied / AskUser
                  └─────────────────┘
```

**关键点**：执行器只能看到布尔值（Allowed/Denied）。权限管理器内部处理三元决策（Allowed/Denied/AskUser），在 AskUser 流程完成后才将最终布尔值返回给执行器。

## 执行器（安全边界）

执行器是 I/O 操作的**唯一通道**。它们强制执行：

### 独立调度线程

每个执行器拥有**独立的调度线程**：

- 执行器之间的线程相互隔离——一个执行器的线程阻塞不影响其他执行器。
- 每个执行器可以设置独立的资源限制（CPU、内存等）。
- 使用线程池技术管理执行器线程。

### 请求队列

每个执行器维护一个请求队列：

- 请求按类型路由到对应的执行器。
- 支持优先级排队。
- 每个请求有超时控制。

### 权限验证的线程锁定

当工具发起资源访问时：

1. 执行器接收请求并**锁定当前线程**。
2. 执行器查询该硅基人的私有权限管理器。
3. 如果回调返回 AskUser，执行器线程**保持锁定**等待用户响应。
4. 硅基人只能看到最终结果（成功或拒绝）——看不到中间的"挂起"或"等待"状态。
5. **只有硅基主理人**会触发真正的用户询问。普通硅基人通过全局 ACL 同步查表，不产生阻塞。
6. 超时后请求视为拒绝，释放线程锁定。

### 执行器类型

| 执行器 | 范围 | 默认超时 |
|--------|------|----------|
| `DiskExecutor` | 文件读写、目录操作 | 30秒 |
| `NetworkExecutor` | HTTP 请求、WebSocket 连接 | 60秒 |
| `CommandLineExecutor` | Shell 命令执行 | 120秒 |
| `DynamicCompilationExecutor` | Roslyn 内存编译 | 60秒 |

### 异常隔离与容错

- 一个执行器的异常不影响其他执行器。
- 线程崩溃后自动重启。
- 熔断机制：连续失败时暂时熔断，防止雪崩。

## 全局 ACL（访问控制列表）

持久化到存储的共享规则表，**仅由硅基主理人管理**：

```json
{
  "rules": [
    { "prefix": "network:api.github.com", "result": "Allowed" },
    { "prefix": "file:C:\\Windows", "result": "Denied" },
    { "prefix": "cli:rm -rf", "result": "Denied" }
  ]
}
```

- 规则按顺序评估，首次匹配生效。
- 只有硅基主理人能修改全局 ACL（通过其专用工具）。
- 修改立即生效。
- 全局 ACL **不在**上述每次查询的优先级链路中——它由回调函数内部引用。

## 用户高频缓存

为减少重复的权限提示，系统为每个硅基人维护两个**仅存在于内存中**的缓存：

| 缓存 | 用途 |
|------|------|
| **高频允许（HighAllow）** | 用户频繁允许的资源 |
| **高频拒绝（HighDeny）** | 用户频繁拒绝的资源 |

### 工作机制

- **用户主动选择加入，非系统自动检测**：当 AskUser 触发时，用户选择是否将资源加入缓存。
- **前缀匹配**：支持资源路径的前缀匹配（如 `network:api.example.com/*`）。
- **优先级**：高频拒绝优先级高于高频允许。
- **仅内存态**：缓存不持久化，重启后丢失。
- **可配置有效期**：用户可设置缓存条目的有效时长。

### 缓存更新流程

1. 权限回调返回 `AskUser`。
2. 权限系统向卡片系统（Web UI 或 IM）发送询问。
3. 用户做出决策（允许/拒绝）并**选择是否加入缓存**。
4. 卡片系统返回决策 + 缓存标记。
5. 权限系统更新对应的缓存列表。
6. 后续匹配缓存前缀的请求将立即解析。

## 用户询问机制

当权限检查返回 `AskUser` 时：

### Web UI：交互卡片

网页前端立即弹出**交互卡片**，显示：

- 资源类型和路径
- 操作描述
- 允许 / 拒绝按钮
- 可选的"始终允许" / "始终拒绝"复选框（加入高频缓存）

### IM（不支持卡片）：随机码

对于不支持交互卡片的即时通讯平台：

1. 系统生成两个随机 6 位数字：**允许码**和**拒绝码**。
2. 发送包含资源信息和两个码的消息。
3. 用户必须回复精确的允许码才能授权。其他任何回复均视为拒绝。
4. 码为一次性使用，防止重放攻击。

### 超时

- 所有 AskUser 请求都设有超时时间。
- 超时后请求视为**拒绝**，释放执行器线程锁定。

## 动态编译安全

自我进化（类重写）引入了独特的安全风险。系统通过**分层策略**缓解：

### 第一层：编译时引用控制（主要防线）

- 编译器只获得**允许的程序集引用列表**。
- **允许**：`System.Runtime`、`System.Private.CoreLib`、项目程序集（ITool 接口等）
- **禁止**：`System.IO`、`System.Reflection`、`System.Runtime.InteropServices` 等
- 代码中引用了禁止的程序集时，**编译器本身会报错并拒绝编译**
- 这比运行时扫描更可靠——危险操作在类型层面就不可能存在

### 第二层：运行时静态分析（辅助防线）

- 即使编译成功，仍对代码进行静态模式扫描。
- 检测直接 I/O 访问、系统调用等危险操作模式。
- 发现危险代码时拒绝加载并回退到默认函数。

### 继承约束

所有自定义硅基人类**必须**继承 `SiliconBeingBase`。编译器在类型层面强制执行此约束。

### 加密存储

编译后的代码以 AES-256 加密形式存储在磁盘上：

- **密钥派生**：通过 PBKDF2 从硅基人 GUID（全大写）派生。
- **解密失败**：回退到默认实现。
- **运行时重编译**：新代码先在内存中编译；仅在编译成功且实例替换后才持久化。

### 原子替换

替换过程是原子的：

1. 在内存中编译新代码 → 获得 `Type`。
2. 从 `Type` 创建新实例。
3. 将旧实例状态迁移到新实例。
4. 交换引用。
5. 持久化加密代码。

任何步骤失败，旧实例保持活跃。

## 权限回调函数

### 设计

每个权限管理器持有一个**回调函数变量**：

- **默认**：指向内置的默认权限函数。
- **动态编译后**：被硅基人的自定义权限函数覆盖。
- **二选一**：同一时刻只有一个回调生效。
- **编译失败不影响**：保持当前生效的函数（默认函数或上一次编译成功的自定义函数）。

### 回调签名

```
PermissionResult Callback(PermissionType type, string resourcePath, Guid callerId)
```

返回 `Allowed`、`Denied` 或 `AskUser`。

## 审计日志

所有权限决策都会被记录：

```
[2026-04-01 15:30:25] ALLOWED  | 硅基人:助手A | 类型:NetworkAccess | 资源:api.github.com | 来源:高频允许缓存
[2026-04-01 15:30:26] DENIED   | 硅基人:助手B | 类型:FileAccess    | 资源:C:\Windows\System32 | 来源:高频拒绝缓存
[2026-04-01 15:30:27] ASK_USER | 硅基人:主理人 | 类型:CommandLine   | 资源:del /f /q *.log | 来源:回调函数
[2026-04-01 15:30:28] ALLOWED  | 硅基人:主理人 | 类型:CommandLine   | 资源:del /f /q *.log | 来源:用户决策
```

日志持久化到存储，可通过 Web 界面（日志控制器）查看。
